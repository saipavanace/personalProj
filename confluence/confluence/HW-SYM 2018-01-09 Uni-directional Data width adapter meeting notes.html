60 minData width adapter microarchitecture reviewShailendra Aulakh (Deactivated)John Coddington (Deactivated)
  • Following points were discussed
    1. Architecture should support HP = 1 to HP = 0 and vice versa or HS to HP and HP to HS. Should be able to have one architecture with certain logic optimized according to HP conditions for input and output links.
    2. N (ratio of field widths in chunks i.e. input link width: output link width) could be different for Header and Data fields. And architecture should support this.
    3. When N (ratio of field width in chunks) varies from field to field in a Link packet definition, error out with a message.
    4. Explore datapath changes with a Pipeline-based structure (de-packetizer/SIPO) possibly using sym_pipe so as to have one generic structure that doesn’t change a whole lot for when conditions such as HP, or widths change.
    5. Current control logic won't change much with above changes. Will continue to have counters to count ingress and egress phits, one of the counters counting entire header length.
    6. The 'last' signal can be used to detect end of the data payload and not necessary to count the entire payload size of phits.
    7. Currently assuming Endianness is constant in a fabric. Need to Check with Monica.
    8. Add CDW support section TBD.
    9. Protection bits. Narrow to wide – Need to consider protection for when packet terminates on an odd data beat, resulting mismatch in the number of data bytes on ingress and egress link. If protection scheme remains constant, then cloning the last byte to the remaining beats of the output phit, might take care of both ECC and Parity schemes.
