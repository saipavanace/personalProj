Documentation Process OverviewPerformed

The process is well defined for all customer documenation

  
Document RequestDefinedKeeps track of all documentation requests, Ensures that doc numbers don't get duplicatedSMEs need to use this first step of the process. It's only been used for PIANO1. 
Design and PlanDefinedThe process is well defined and plans are followed after discussions with SME's on document delivery.SMEs do not have a good idea on the type of documents required for customers. The timing of the request is off where the technical writer does not have enough time to produce good quality documentation. 
Content DevelopmentDefinedThis process involves collecting information from SMEs, organizing the content.Content is not provided in a timely manner. Most of the time the content is unavailable in documents. Illustrations are on a white board or hand drawn. The technical writer has to go back and forth for review. Engineering specifications lacks illustrations, however, customer documentation requires illustrations. Internal documentation is not available to create customer docs especially for software. 
Review/ApprovalDefined/PerformedThis process allows SMEs to review the content, make changes and provide feedback to the Technical Writer.Reviewers usually do not provide feedback on time. SMEs need to do a thorough review of documentation to ensure that correct information is captured in customer documents. Documentation must be reported on JIRA. Only after the final review and feedback, online help is generated. 
Document ReleaseDefinedDocumentation is release to customers.After all documentation is approved, final documentation is released. However, there is a sign-off process where SMEs have to sign-off on the final release. It's only been done once for the PIANO 1.0 release. 
Document ArchiveDefined/PerformedDocumentation for every version of the product is archived.After every release, documentation associated with the particular release is archived and stored on SVN.
