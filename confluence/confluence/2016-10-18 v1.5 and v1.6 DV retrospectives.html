Discussion items

TimeItemWhoNotes
Didnt go well
  • Thought that some things were done, but they were not actually done. That came out with hashtag implementation. Less amount of feedback/review for DV. (P1)
  • No latency/throughput requirement defined up front. That caused some more churn. (P1)
  • Test planning did not go well. Hash tags solved some of this, but we need multiple test plan reviews and we should update test plans as soon as holes are discovered in the TB. Holes/missing test functions did not get accounted for in schedule. (P1)
  • Hash tags should have percentage done tracking and how much effort they will take. (P3)
  • Cherry picking because multiple projects are being done in parallel. Pre-push should only run on a check-in that has been published. Come up with a way of formally doing cherry-picking (maybe always check-in into multiple branches when making a change instead of doing it on one branch first and another branch later) (P2)
  • Be clear with where a hash tag comes from and what its scope is more precisely. (P2)
  • No gap between RTL freeze and DV closure. (P1)
  • Software codegen had a lot of issues and that delayed system hw. Very difficult to debug codegen failures. Rolling back codegen is not an option. Codegen should go through hw release mechanism so that dv/hw should not face any issues. Possibly bring back bamboo for codegen updates (maybe just a small set of tests). Once codegen is changed, an automated script should update refresh_hw/setup_hw scripts. (P1)
  • Error/transport errors should be done in hw while writing RTL features instead of the end. Wont be able to verify errors/csrs if all these features are coded at the end of a project. RTL feature complete should be a lot earlier than project release date. (P1)
  • RTL feature complete date should be a lot earlier than release date. No time for DV to finish up if there is not enough time at the end. Add milestones. (P1)
  • CSR tests and errors in v1.5 was not complete, causing lots of bugs to be found in v1.6. Didnt enough time/framework to get things done in v1.5. Common framework missing in places where they can be used (for ex: pressure implementation) (P1)
  • Schedule should include last minute timing and last minute bug fixes. Projects are getting delayed because these issues are not being taken into account up front. Risk assessment should be done on these last minute changes to gauge how much schedule will slip due to them. (P1)
  • Duplication of work because multiple projects with release dates close to each other. Come up with a regular cadence. So that we dont have to run coverage and customer TB so often. (P1)
  • For every release, there could be features that could be left out if the feature is not complete. So we don't miss a release cadence. (P1)
  • Managers should not have so many IC tasks on themselves. Focus will then be on tasks and not on schedule tracking. (P2)
  • Hiring and attrition. (P1)
  • Mon/Thu meetings should be more issue focused (Are you stuck on something? v/s Where are you on each task?). Maybe use one meeting for schedule updates and the other to unblock people and figure out issues that are coming up. (P2)
 Went well 
  • Hash tags went really well. Clear scope of what needs to be done.
  • Cron job for regressions.
  • Delays were not as bad as v1.0.
  • Confluence has been very good for communication.
  • v1.6 was on time.
  • Most of codegen related stuff has become better from v1.0 to v1.5. There is still scope for improvement so that codegen updates are transparent to dv/hw.
  • We are good team. Everyone has been here for a while.
