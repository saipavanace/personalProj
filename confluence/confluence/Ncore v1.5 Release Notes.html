Release Notes / Arteris Product Suite / v1.5

What's New

The Ncore v1.5 release includes the following enhancements and changes:

  • Increased parameter ranges, allowing for larger and more sphisticated coherence subsystem designs.  The parameter range changes include:

    • up to 6 coherent memory interface ports

    • up to 8 fully-coherent agent interface ports 

    • up to 4 IO-coherent agent interface ports 

    • up to 4 non-coherent bridge ports 

    • up to 4 directory ports 

    • up to 6 snoop filters (1-16 ways)

    • up to 4-way interface interleaving of interfaces

    • up to 1024 sets by up to 16 ways of proxy cache per NCB port

    • 8 to 96 outstanding protocol transactions per coherent agent interface port

    • 8 to 96 active protocol transactions per directory port

  • Coherence unit interfaces have configurable data widths:  64 bit, 128 bit, or 256 bit.
  • Agent Isolation Mode, providing a low-power mode when only one agent is operating. See <RM reference>.
  • Optional snoop filter victim buffer for improved performance.  See <RM reference>.
  • Optional support for multiple outstanding transactions per transaction ID.
  • Optional silicon debug features.  See <RM reference>.
  • Support for SystemC simulation models, and Synopsys Platform Architect integration.
  • Support for cadence simulation tools.
  • Various area optimizations.
  • Significant changes to how clocks, resets, power domains, and voltage regions

Design Migration

Because of the significant changes to clocks, resets, power domains, and voltage regions, designs from a previous version of Ncore cannot be automatically migrated to Ncore v1.5.  When you open a project from a previous Ncore version, you will be shown instructions for removing the transport interconnect and clocks portion of the previous design, while keeping the coherent subsystem portion of the design.  Contact your Arteris Applications Engineer for help migrating your Ncore 1.x design to Ncore 1.5.

Known Issues

 

Conductor Application

  • CONC-1429

CONC-1429: No checking for Reserved keywords  CONC-1429

Assigning a Verilog or System C reserved keyword to the project name causes compilation failures.
Workaround: Do not name projects with reserved keywords.

 

  • COND-557

COND-557: Architecture View settings are not maintained

View settings in the Transport Definition perspective architecture topology editor are not maintained when switching to another section of the transport definition perspective. When the topology editor is next opened, default view will be displayed.

 

  • COND-536

COND-536: Deleting clock manager can cause an internal error

Inside the Transport Definition perspective, deleting a clock manager or clock regime that contains multiple clocks can generate an internal error.
Workaround: This internal error does not affect the stability of the project, as it referenced one of the deleted clocks.

 

  • CONC-2059

CONC-2059: Exchanging the names of concerto units causes an error in the Transport Definition perspective.

Exchanging the name of two concerto units will result in an error when going to the transport interconnect.  For example, renaming "Unit A" to "Unit "B", and renaming "Unit B" to "Unit A" without going to the Transport Definition perspective in between causes the error.

Workaround: Rename the first unit to a temporary name, and go to the Transport Perspective.  Then, make the final desired name changes.

Ncore Interconnect

  • CONC-1043

CONC-1043: Throughput limitation with a small number of sets
Each set in the directory is limited to one transaction every four cycles. If the directory is configured for a single set per port, throughput is limited to one transaction per 4 clock cycles, and if it is configured for 2 sets per port, throughput is limited to one transaction per 2 clock cycles.
Workaround: Configure all tag filters with the number of sets greater than the number of directory ports.

 

  • CONC-1656

CONC-1656: User security flags in the coherent-capable Transport Interconnect are not supported
When using the Coherent-capable Transport Interconnect (CCTI) to carry non-coherent transactions, additional security flags are not supported.
Workaround: Only use security flags already defined for the CCTI, or use a separate Non-Coherent Transport Interconnect for your non-coherent traffic.

 

  • COND-543

COND-543: Lint warnings

Depending on the configuration, a lint tool may report warnings about unconnected signals.

Workaround: These signals are not used by the design, and can be safely ignored.

Performance Modeling

  • COND-567

COND-567: Synopsys SystemC AXI TLM transactors not automated

The SystemC performance modeling environment only instantiates Synopsys SystemC TLM transactors for ACE interfaces.  Other interface types, including AXI, are not instantiated automatically.

Workaround: Instantiate and connect AXI and other interfaces manually.

 

  • COND-566

COND-566: SystemC flow does not support Synopsys memory models

The SystemC performance modeling environment does not support Synopsys compiled memories. If Synopsys compiled memories are used, the performance modeling build environment will produce a non-functional model.

Workaround: Change the Ncore embedded memory implementation to be "Registers".

Example Synthesis Scripts

  • CONC-1706

CONC-1706: Improperly constrained asynchronous bridge
When using Transaction Probes and Using Resilience Aggregators across asynchronous clock domains, the example synthesis scripts will have missing constraints for the asynchronous bridge between the probe and the aggregator.
Workaround: Contact your FAE to get the correct synthesis constraints for your design.

 

  • COND-595

COND-595:  Error in Clock Enable input_delay

When using a clock with a clock enable port, the input_delay for the clock enable port will be set using a non-existing environment variable.
Workaround: Fix the input_delay line at the top of optl.tcl file by replacing the non-existent environment variable with the correct one.

 

Example Testbench

  • COND-601

COND-601: Customer test bench only supports AXI4 non-coherent interfaces.

When the non-coherent transactions for Memory Interfaces, Agent Interfaces, and Non-Coherent bridges are not exported, but are instead connected internally to a Transport Interconnect (TI), the example test bench instantiates verification IP for AXI4 interfaces (other than  read-only or write-only interfaces) that have connectivity to the non-coherent interfaces. Interfaces other than AXI4 have their signals driven to 0, which will cause the test bench to fail. 

 

Additional Information

  •   CONC-1989

Verification IP does not support wAxId of 0
The Synopsys and Cadence Verification IP used by the example test bench do not support an AxID width of zero.

 

  • Platform Architect supports pin accurate transactors only for AXI4 interfaces with the default widths for all signals except AxADDR, xDATA, and AxID.  Not using the default width for other signals will prevent you from being able to use the Platform Architect PinAccurate converter.

COND-566           : SystemC flow does not convert or uses Synopsys memory models

If a Ncore instance uses Synopsys memory models to implement its buffers as SRAM, then the flow to convert the Verilog to SystemC will produce non-functional SystemC result.

Workaround: change the Ncore buffer implementation to Registers (Generation / Embedded Memories / Memory type = Registers).
