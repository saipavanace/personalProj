1. Introduction

This document is an attempt to gather industry standard coding guidelines under one roof to have a code base which is easier to read, understand, maintain and debug. Until all the below suggestions are agreed upon, this page might be continuously evolving 

2. TACHL Code vs Generated Code

  • We should make an effort to organize our source code and at ARTERIS our source code is TACHL (or system verilog with embedded JS). 
  • Pros of organizing TACHL code
    • This is the code we write and hence easier to organize
    • Less stress to organize TACHL code as we don't have to think about how the written code would end up in our generated code
    • It is not always easy to write the TACHL code in a way that the generated file is properly organized
    • It is easy to fold a block of code in tools like VS code and thus navigate through the code faster when everything is properly indented
  • Cons of organizing TACHL code
    • Generated system verilog code is disorganized. 

3. Indentation

  • Use 4 spaces of indentation at the beginning of any code blocks (like if, for, begin, while, repeat, etc)

       BAD

if(<expression>) begin
my_statement1;
  my_statement2;
end

       GOOD

if(<expression>) begin
    my_statement1;
    my_statement2;
end


  • We use Embedded Javascript in our source code and so philosophically, javascript is also part of our source code. Hence tokens like "<%if", "<%for", "%>" need to be considered as our keywords and should be appropriately indented

       BAD

if(<expression>) begin
<%if(<expression>){%>
    my_statement1;                   
<%}%>
end

       GOOD

if(<expression>) begin
    <%if(<expression>){%>
        my_statement1;                   
    <%}%>
end
  • DO NOT use TAB to maintain consistency between editors.
  • If you want to use tab, convert tab space to 4 spaces like below:

      In VIM:

" place this in ~/.vimrc
set tabstop=4
set shiftwidth=4
set expandtab


      In EMACS:

; place this in ~/.emacs
; http://stackoverflow.com/questions/69934/set-4-space-indent-in-emacs-in-text-mode
(setq-default indent-tabs-mode nil)
(setq-default tab-width 4)
(setq indent-line-function 'insert-tab)

      In VS Code:  

  • Go To :
    • File
    • Preferences
    • Settings
    • Search for tabSize
    • Uncheck "Editor: Detect Indentation"
    • enter value 4 for "Editor: Tab Size"

4. Functions

  • If a function (or task) have a long list of arguments, we should write each operand on a new line starting from the second argument which is aligned to the delimiter that denotes the start of the function (or task)
  • Each argument must be explicitly typed

       BAD

  function <return_type> function_name( bit a, bit b, bit c, d);
      // function body
  endfunction : function_name

       GOOD

function <return_type> function_name(bit a, 
                                     bit b, 
                                     bit c, 
                                     bit d);
    // function body
endfunction : function_name
  • We should break larger functions (or tasks) to smaller functions (or tasks) which can be easily understood with descriptive names

5. Indenting expression variables in "if" condition

  • When a "if" condition expression have more than 2 operands, we need to write each operand on a new line starting with the second operand which is aligned to the delimiter of the "if" condition
  • Also the operator in between the operands need to be placed at the end of the operand on each line

       BAD

    if(a && b && c && d) begin
      //if block
      
    end

       GOOD

    if(a && 
       b && 
       c && 
       d) begin
      //if block
    end
  • If the expression gets very complicated such that it is not trivial to align the operands, then we need to use a function with a descriptive name and align the arguments of the functions with each argument in a new line starting with the second argument

6. Searching Queues

  • When search for an item in a queue, when the search condition gets very long, we should use a function with a descriptive name which evaluates the search condition and pass it back to the queue

       BAD

 test_q = my_q.find_index with (item.val1 == 1 &&
                                item.val2 == 5 &&
                                item.val3 > 5  &&
                                item.val4 ? 1 : 0);

       GOOD

 test_q = my_q.find_index with (item_in_range(item));

 //Function Definition
 function bit item_in_range(item_type passed_item) begin   // The if condition below can be more explicitly written for better debugging and also include debug statements
    if(passed_item.val1 == 1 &&
       passed_item.val2 == 5 &&
       passed_item.val3 > 5) begin

       if(passed_item.val4) begin
           return 1;
       end else begin
           return 0;
       end
    end else begin
       return 0;
    end
 endfunction : item_in_range

7. Begin/End

  • begin goes on the same line as the first statement of the block it belongs to

       BAD

  if(<expression>)
  begin
      my_statement1;
  end

       GOOD

  if(<expression>) begin
      my_statement1;
  end
  • If there is an else part, end - else - begin go on the same line for the else block

       BAD

  if(<expression>) begin
      my_statement1;
  end
  else begin
      my_statement2;
  end

       GOOD

  if(<expression>) begin
      my_statement1;
  end else begin
      my_statement2;
  end
  • Always wrap an if and else blocks in begin and end statements even if they contain a single line

       BAD

  if(<expression>)
      my_statement1;
  else
      my_statement2;

       GOOD

  if(<expression>) begin
      my_statement1;
  end else begin
      my_statement2;
  end

8. Labelling Code Blocks

  • We should label the outer code statement starting with 'begin' and 'end' with descriptive label names so that it gives as a sense of where the code block beings and where it ends

      BAD

  if(<expression>) begin
      my_statement1;
  end else begin
      my_statement2;
  end

      GOOD

  if(<expression>) begin : my_if_block
      my_statement1;
  end else begin
      my_statement2;
  end : my_if_block
  • We should add label with ":" token instead of writing them as comments because when we use the ":", the tool can pick up on mismatches for the labels which might help us avoid issues later.

9. Comments

  • Every file should have a copyright notice at the top
  • Every file should have a header describing the file following the copyright notice
  • Every class definition should have comments to describe the class above it's definition including the public variables and methods
  • Every function definition should have comments to describe the function above it's definition including arguments and return values
  • Any code that is commented out should accompany with an additional comment describing why it was not deleted (prefer deletion)
  • Any code that needs to be revisited should have "TODO:" token (which can be parsed with a tool later) to be present in the comments

11. Other Recommendations

11.1 Using Plusargs

  • Use "plusargs" as minimally as possible as they reduce the code maintainability and debuggability.
  • These constructs slow down the simulation
  • See where all we can override the classes from extending from them when we want to change the behavior of it instead of using a plusarg. Sure we will have to spend more time to extend and create this class but this will be a one time investment.

11.2 Using uvm_objections

  • Do not raise objections and drop objections when not needed as they have severe performance penalty
  • There should ideally be only a single objection raised in the entire testbench.
  •  Use counters or another tracking method where needed to determine that a sequence has truly completed
    • Second objection raised in a virtual sequence is acceptable if it is absolutely necessary (in case we want to wait until all transactions are complete). But anything more than that is not acceptable.

LOW PERFORMANCE CODE

class sequence extends uvm_sequence#(seq_item);
    //....................
    task body;
        uvm_objection objection = new("objection");
        seq_item item = seq_item::type_id::create("item");
        repeat(5) begin
	        start_item(item);
	        assert(item.randomize());
       	    objection.raise_objection(this);
	        finish_item(item);
	        objection.drop_objection(this);
        end
    endtask
endclass

//..........
task body;
    sequence seq = sequence::type_id::create("seq");
    seq.start(seqr);
endtask

HIGH PERFORMANCE CODE

class sequence extends uvm_sequence#(seq_item);
    //....................
    task body;
        uvm_objection objection = new("objection");
        seq_item item = seq_item::type_id::create("item");
        repeat(5) begin
	        start_item(item);
	        assert(item.randomize());
	        finish_item(item);
        end
    endtask
endclass

//..........
task body;
    sequence seq = sequence::type_id::create("seq");
    objection.raise_objection(this);
    seq.start(seqr);
    objection.drop_objection(this);
endtask

11.3 Using config_db:

  • Using config_db also comes at a severe performance penalty and it also slows down the debug process
  • The more entries we add to the config_db, the more time it takes for the simulator to search all the entries to find the entry we are looking for, and thus slower the entire simulation will get
  • We should ideally use config_db only to pass the virtual interface handles. If we are using it to pass handles or variables from one class to another, there might be room to improve the structure of the testbench
  • During unforeseen scenarios, where we must use the config_db, we should bag all the variables in a singleton class and assign the handle to this object to config_db instead.

11.4 File naming

  • always use .svh as the extension for header or include files (specifically, avoid using .svi)
  • interfaces should be named using the suffix _if.sv for example my_great_interface_if.sv
