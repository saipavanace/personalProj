WORK IN PROGRESS

Before we get deep into Symphony development, we have an opportunity to re-examine some of our decisions on how we do the software development for Symphony.

  • General programming
    • Validity check all user input
    • Check return codes for anything returning codes, try/catch for anything that could throw exceptions
    • Short, modular routines
    • Meaningful variable, method, and class names
    • Unit tests + test coverage
    • System and flow tests

  • Operating Systems
    • We need to support RHEL 6.6+, RHEL 7.x, SLES 11.x and SLES 12.x
    • Possible choice: do our work on CentOS 6.6, build on CentOS 6.6, and test on the rest.
      • Advantage: our work would be done in the same base environment that we ship
        This assumes that we can ship a single build and it will work on multiple opsys versions.
    • Other possibility: do our work on CentOS 7.x, build on CentOS 6.6 and test on the rest.
      • Advantage: some tools are not available on CentOS 6 (e.g. atom)
  • Languages
    • Javascript - what version of Javascript will we use for our scripting needs?
      • Version of ECMAScript
        • ECMScript 5th Edition
        • ECMAScript 2015 (A.K.A. ECMAScript 6th Edition)
        • ECMAScript 2016 (A.K.A. ECMAScript 7th Edition)
        • ECMAScript 2017 - Feature set is complete
        • ECMAScript 2018 - Still being defined
      • IDEs - Should we specify an IDE or IDEs for sw work?  If so, which one(s)?
        • Atom (not available on Centos 6.6+)
        • WebStorm
        • Eclipse
        • text editor support: Vim, Sublime, Emacs
      • Lint
        • ESlint
          • Need to have eslint shareable config for company-wide required checks
        • Any other linters we should evaluate?
      • Type Checking
        • Flow static type checker
      • Testing
        • Unit tests
          • Mocha
          • Others?
        • System tests
        • GUI tests
          • Sikuli
    • TCL - what version of TCL will we use for our TCL interface and customer scripting needs?
      • What version is supported by the EDA tools?  We should be as compatible as possible.
    • C++ - what version of the C++ compiler will we target for our C++ needs?
      • C++98 (C++ 1998/2003 Standard)
      • C++11 (C++ 2011 Standard)
      • C++14 (C++ 2014 Standard)

  • GUI development environment
    • This depends heavily on our choice of GUI for Symphony
      • Nw.js vs browser-based vs Qt
      • React vs Angular vs ...

  • Source code control
    • Any reason not to continue using Git and Stash?

  • Continuous Build
    • Bamboo seems to be working OK, so is there any need to examine other options?

  • Packaging
    • Javascript
      • Browserify
      • V8 compiled code?
      • Web Assembly?  Is it a viable option right now?

  • Libraries
    • Message handling (message dictionaries?)
    • Error/warning handling
    • User configuration save/restore
      • per-directory, per-user, per-project, per-site

  • Process
    • Agile - scrum, sprints, etc - what works, what doesn't?
    • Tracking with JIRA - what works, what doesn't?
