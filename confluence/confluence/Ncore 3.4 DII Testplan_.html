Introduction

DII (Distributed IO Interface) is an Ncore module at the lower boundary of Ncore.

DII accesses noncoherent address space at the request of Ncore AIUs (Agent Interface Units).

DII additionally supports

  • device memory which may be volatile or have side effects (such as config registers of an IP)

  • well behaved memory without side effects (such as DRAM)

  • Ncore transaction ordering modes

  • error tolerance, detection and reporting


A minimal instance of DII is used to access Configuration and Status Registers of Ncore units (CSR). This instance has 32-bit AXI native interface, and supports only 32-bit aligned device non-modifiable non-buffered read/write accesses.

Assumptions

DII::

-- only AIU initiates

– smi initiator <-> axi endpoint

-- noncoherent

-- no internal cache

-- ordering model: CHI ordering (csymlayers:Layers); AXI ordering

-- error model: sram random faults, transport error signal, propagate external errors identified within protocols.

   transport random faults are covered by resilience feature separate from dii.

-- 32b Dii shall service only 4B, size aligned accesses.   CONC-5031 - specify csr-dii cmdset CLOSED

List and description of TB components

             Dii TB is based around searchable lists of object Dii_txn, which represents the end to end journey of a single request from upstream.

             Data manipulation methods are single-sourced at their maximum level of abstraction, mainly in Dii_txn or Dii_txn_q.

             Any existing data obj is guaranteed to contain meaningful information.  access to data not yet present is a null dereference.  This is an intentional mechanism for preventing silent corruption when a component is not ready for use case (due to development state, or mistake) .

Transaction

             State machine representing the end to end journey of a single request from upstream.

             most of the checks in this doc are implemented here.

             collects smi_seq_item, axi*_seq_item representing each msg in the txn.

             api gen_expd__<Msgtype>() constructs the fields in a next msg of <Msgtype> which may be derived from the current state of the txn.

Transaction queue

                     data store of txns.

                     contains certain methods operating across multiple txns, such as ordering checks.

Checker

            Objective: die iff any mismatch with srch.  Error message sufficient to debug without wave.

            maintains txn_q of all outstanding Dii_txn.

            maintains order_q of all Dii_txn which have not passed their ordering point.

            Upon receiving a msg, generates and compares to the msg of that type which is expected by the corresponding txn.

            Internally sequences messages of an axi txn to correlate with dii_txn.

            input: smi_seq_item, axi*_seq_item observed by monitors

Coverage

           Objective: cover all functions and corner cases of dii.

           concerto: ${WORK_TOP)/dv/common/lib_tb/smi_coverage.svh 

           axi : ${WORK_TOP)/dv/dii/env/dii_coverage.svh

           dii:  ${WORK_TOP)/dv/dii/env/dii_coverage.svh

BFMs 

concerto

            Objective:  generate all concerto traffic which dii may receive.

            knobs adjust distribution of traffic.

            knobs enable traffic not expected during normal operation, such as uncorrectable errors.

            txn based seq which generates all possible next msgs for a txn

            use systemverilog constrained random

axi

           Reactive bfm.   

           Read returns random data.  not a memory model.

           knobs enable error injection.

apb

            For register access (CSR).

            We access implicitly via reg,field name apis in mentor RAL.


Monitors

            Sends msg to scoreboard once all beats of data have been collected.

smi

           Collects data on interface and creates smi_seq_item.

           common monitor

axi

           Collects data on interface and creates various axi seq_items.

PMON uArchitecture :

Pmon Presentation :

The performance counter unit can track and report upto 31 events. The Ncore unit instantiating the performance unit
specifies the events. These events maybe single bit or multi bit events. The performance counter unit can be configured to implement either 4 or 8 counters.
Each counter is implemented as a 64 bit counter that is capable of counting upto 2 events at a time with following
counter modes : 

1. Normal count: In this mode the counter counts both the events together, if both the events are asserted then
it is counted as two, if one event is asserted then it is counted as one
2. AND count: In this mode the counter counts one if both the events are asserted.
3. XOR count: In this mode the counter counts one only if one event is asserted and the other is de asserted.
4. Instantaneous count: In this mode the counter provides the instantaneous value of the multi bit event as is.
5. 32-bit count: In this mode the 64 bit counter acts as two 32 bit counters for two separate events

The counter reports count via 2 registers one is a 32-bit count register that reports lower 32 bits and another configurable
register which supports
1. Capturing the upper 32 bits of the count
2. Used as an accumulation register for IIR filter
3. Use as max/saturation value for accumulation events
4. Use as 32 bit counter for second event

Pmon new features (Ncore3.4) : 

Latency counter :

Ncore provides latency counters DII for either reads or writes. Latency is reported back as a binned
histogram. The latency reported at DII is more of a closer representation of latency seen at the Native AXI interface.

When transaction is ready to be issued on the Native interface, it is allocated to the latency counter table.

The ID represents the transaction table (OTT/WTT/RTT) ID and counter is a 9-bit counter. The counter is started as soon as the entry is allocated. If all entries within the latency counter table are taken, then any new qualified transaction just does not participate in the latency histogram.

The counter increments every 2, 4, 8 or 16 clock cycle based on CSR configuration. If the counter hits its max value, then it stays saturated until deallocation and the saturated value is used for latency histogram binning.

The latency counter table is deallocated at approximately the same time corresponding response is received from the native interface. The deallocated entry counter value is then subtracted by the configured 8-bit offset, if the offset corrected value is negative then it is saturated downwards to zero. The quantizer takes the offset corrected value and increment appropriate histogram bin.


Latency counter verification strategy

The latency table verification strategy consists of spying on the "alloc" and "dealloc" signals with a latency spy interface.
These signals will be the inputs of the latency counter scoreboard.


Latency scoreboard predicts and generates the different histogram bins
After that, Those bins will be sent to the Pmon scoreboard which will increment the appropriate bin and map each counting value to the correct register.
The 8 registers are checked at the Pmon scoreboard level.


In order to improve the quality of our verification and to be able to precisely identify and determine the location of any bugs, the latency table verification would be on two levels:

1 - The verification of the generation of the bins to detect a possible bug at the level of the latency counter table or offset logic  : we will spy the bins signals of design at the output of the quantizer and will be compared with the bins generated by latency counter scoreboard at the level of this scoreboard.
2- Verification of the bins counting and mapping of values to the different registers at Pmon scoreboard level.




Bandwidth counters

Ncore provides Bandwidth counters DIIs. These counters count in 64Byte data accuracy and at divided by 16 accuracies of the clock frequency.

BW counting can be filtered based on FunitID or user bits, this can be configured in CSRs xBCNTFR and xBCNTMR.

In DIIs the count following : 
• Read data bandwidth, this refers to data being read from DII : DtrReq event
• Write data bandwidth, this refers to data being written into DII : DtwReq event

The counter reports : 
• xCNTSR: number of divide by 16 clock cycles
• xCNTVR: number of 64Bytes of data.
The above information can be used to calculate the effective BW. The counter must be disabled before reading them to get correct BW.
Note that this will be approximate BW as all transactions that were counted may necessarily not be 64 bytes transactions and the time accuracy is divided by 16.

Trace and capture features : 

TCAP checking: Trace Capture Scoreboard checking

TCAP scoreboard integration

Each unit environment instantiates the Trace Capture Scoreboard.

The unit environment connects the SMI analysis ports to the Trace Capture Scoreboard.

The unit environment writes to the trace capture scoreboard static register fields whenever the Capture Control Register is written.

Once the Trace Capture Scoreboard knows the Capture Control Register value and has the SMI analysis port connections it can perform some checking of the actual DTWs output by the Trace Capture block versus what the scoreboard decides are the trace messages that should be seen in the DTWs.

TCAP strategy

The primary test bench for testing trace capture will be DMI. The other benches: DII, CHI, and IOAIU will be primarily concerned with checking trace capture connectivity in those units, not testing the internals of the trace capture logic.

TCAP Dropped trace messages

If the capture system is full including the SMI network input FIFOs, new SMI messages with traceme set will not lead to trace messages that end being output as DTWs. These are dropped trace messages. RTL will use the performance counters to report how many trace messages are dropped, but not which exact trace messages are dropped. At the end of the simulation, the DMI test bench TCAP scoreboards will compare the drop count reported by RTL and report an error if it is different than the DV drop count, but not check which messages should have been dropped. DV will not determine whether it was appropriate for the RTL to drop a message. DV does not model the RTL trace FIFOs, accumulators and buffer. 

Trace messages should not be stuck in the TCAP accumulators or trace buffer at the end of a simulation because the CHI, IOAIU, DMI and DII blocks wait for RTL to be not busy (including trace messages that have not been output) before ending the simulation. DV compares the number of trace messages processed with the number that RTL reports as completed, so we should report an error if any trace message were still stuck in TCAP at the end of the simulation.

TCAP Timestamp checking


Even within a single DTW, timestamp values are not guaranteed to be increasing, they can go up or down, unless only trace capture from one SMI network is specified in the Capture Control Register.

We do the following TCAP timestamp check.

The check has a timestamp check interval specified by +tStampChk=2000 (current default value).
Once every 2000 Trace Messages from DTWs are observed, checker does the following.
Compares the average of the timestamps in the current Trace Message, to the average of the timestamps in the Trace Message from 2000 Trace Messages ago (old_Trace_Message_timestamp_average).
Basically, we check that Trace_Message_timestamp_average(n) > Trace_Message_timestamp_average(n-2000).
The new Trace Message timestamp average must be higher than the old Trace Message timestamp average.
We don’t check the timestamp values for Trace Messages 2001-3999, 4001-5999, etc, just 2000, 4000, 6000, etc.
This is clearly not very “rigorous” checking.

See CONC-8090 for a block level test that fails with +tStampChk=1601.

TCAP Timestamp rollover testing

In the DMI test bench, in some tests DV will force a non-zero initial value for the RTL timestamp value, often to a value, such as 'hFFFF_FF00, that will soon cause a timestamp rollover.

Uncorrected trace

For Ncore 3.2 release, the design create trace messages from the uncorrected SMI packet, not the corrected SMI packet. So, this is what DV will checks. See CONC-8091.

Other checks


The trace scoreboard can do some minimal checking of the legality of some DTW fields and which SMI can be used for DTW output.
