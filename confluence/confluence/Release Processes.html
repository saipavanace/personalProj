The release process covers the processes required to create Release Candidates (RC) and the Final Release.

The RC is created by the engineering team and handed over to the AEs for QA testing once the following goals are met.

  • All required functionality and performance has been completely verified
  • All Timing goals are at 95%. The 5% fall out must have been analyzed and fixes should not result in any additional latency.   
  • All regressions must have at-least 95% pass rate. The 5% fall out must have been debugged and identified as non RTL issues
  • There should not be any open RTL bugs and all resolved bugs should be closed
  • Code coverage must be at-least at  90% of the target coverage rate.
  • Functional coverage must be at-least at 80% of the target coverage rate.
  • All GUI features and description should be verified, reviewed and approved by all stake holders 

RC release should follow the following process:

  • HW team releases, qualified HW to the SW team 
  • SW team builds an RC called RC<n>-build<m>. The steps are described at: SW Ncore Release Processes
  • This RC<n>-build<m> goes through engineering RC release testing: Release Testing
  • Once all the testing passes the  RC<n>-build<m>  is released to AEs as RC<n>

Post RC issue process

Once RC1 is released to the AEs for QA testing any issues found by either engineering or FAE team must follow the following process:

  • All issues should be filed and tracked using JIRA with appropriate severity as described in Jira Process.
  • JIRA filter must be created to track all issues found or fixed after each RC.
  • All issues must have <X>-PossibleReleaseNote label 
  • Issues found by AE team must have QA-<X> label 
  • Issues that affect main line functionality must have MainLineFunc label 
  • All issues must be debugged by the engineering team within a day.
    • Once the issue is debugged the engineering team should provide a recommendation as to when and how the issue should be resolved
    • AE team is required to approve or disapprove the recommendation within a day. Depending on the severity and number of bugs found this can be done over email or a meeting can be scheduled on daily basis      
  • If the issue is resolved before the release <X>-PossibleReleaseNote label is removed
  • If the issue is not resolved then the  <X>-PossibleReleaseNote label is changed to either <X>-Internal_AE or <X>-ReleaseNote and fix version should be updated accordingly. 
    • <X>-Internal_AE: This will be visible to AEs only and they can inform customers on need to basis. Communication text should be provided by the engineering team
    •  <X>-ReleaseNote: This will go in the release notes. Release note text should be provided by the engineering team 
  • If the issue is determined to be invalid  <X>-PossibleReleaseNote label is removed and the issue is resolved as invalid. 

Note that disagreements between AE and Engineering teams regarding when and how issues should be resolved are elevated to Exec staff for resolution. 

After the RC1 has been released a weekly meeting is scheduled between engineering and AE team to track release checklist.

A sample checklist is shown here: v2.0.1 Release checklist

Multiple RCs can be created depending on number of issues found.

The release documentation process is described here : https://arterisip.atlassian.net/wiki/spaces/FIEL/pages/16023773

Once the QA testing, release checklist and documentation process is complete the last RC is converted into the Final Release. 

Post Release Errata Process 

Issues that are found after a release go through the errata process described below:

  • Engineering team labels the bug as <x>-PossibleErrata
  • Once the issue is debugged the Engineering team informs the AE team with following info 
    • possible workaround 
    • All previous releases the issue affects
    • Possible costumers who are affected by the issue
    • Recommendation as to when and how the issue should be resolved
  • AE team analyzes all current costumer configs and identifies who all are affected by the issue
  • Depending on the severity a decision is made regarding the issue either over email or in a meeting. The decision has two parts resolution and communication:
    • Resolution 
      • Wont Fix: This applies if the issue is not fatal/ critical and the workaround is simple enough and acceptable.
      • Fix: This applies if the issue is fatal / critical and the workaround is not acceptable. When and how the fix is made is also decided at this time. 
    • Communication
      • Passive: The bug info is shared internal to the company and the AEs will inform customers on need to basis. The JIRA issue should have following label <X>-Internal_AE
      • Active: An errata release is issued to all active customers. The JIRA issue should have following label <X>-Errata.

Customer Reported Issue Process

Arteris support uses two different issue tracking systems: a) the Arteris support portal, for customer interaction, and b) the Arteris JIRA database for communication with the Arteris engineering team.

Issues that are reported by customers go through the process described below:

  • Customer uses the Arteris Support Portal to fill tickets to report bugs, ask questions, and ask for enhancements. From the portal, they can track progress of their tickets. The portal is also sending automatically emails to customers every time the ticket status changes.
  • Arteris Support Portal Tickets are following the flow shown below (detailed description here):

 

 

  • After the ticket has been filled, an Application Engineer does the initial analysis; and assess if help from Arteris engineering is required. If this is the case, one or multiple tickets are filled in the JIRA engineering database.
  • Each ticket has a severity level as specified in the Jira Process
  • Each ticket is also tagged with labels to identify the customer.
  • The engineering organization follows the Jira Process and makes a recommendation:
  • Not an Issue: The issue comes from an incorrect configuration or incorrect diagnostic from the customer. The Application Engineer communicates this information back to the Customer using the corresponding ticket in the Arteris support portal.
  • Valid Issue will be fixed: Issue reported by the Customer is a real issue and implies potentially a modification of the product. Arteris engineering and Arteris support decide on an appropriate action:
    1. Creation of a patch implying replacement of one or multiple product source files, limited testing, and packaging of a new delivery of the product immediately. This is done of the issue is severe and the customer production flow impacted.
    2. Decision to fix the problem in a future release, already scheduled and part of the roadmap. This is done if the issue is minor, or customer can wait.
    3. In addition, if a workaround exists, this is communicated to the customer.
  • The issue is considered to be definitively fixed once one of the following happen:
    1. the customer did not request a modified version of the product and is satisfied with Arteris response (workaround, recommendation)
    2. the customer required a fixed version of the product, and it has been released.
  • If an issue is severe and is impacting multiple customers, the Errata document shall be updated immediately, and communicated to every affected Customers. Application Engineering is responsible for deciding which customers will be notified.
  • If an issue is not severe, but still affects significantly customers, and it has been decided to fix it in an upcoming release of the product, it is documented in the Errata document for that release.
