45minConfig, debug & trace updateJohn Coddington (Deactivated)
  • Config
    • Hypervisor (highest trust level, has access to everything), debug (a bit more trusted than user), user regions
    • initiator ATU has option to filter error responses (hide the error) - need to decide if it is design-time decision or SW programmable
    • can have multiple config networks; can have firewalls on config networks; can have parity/ECC options
    • config networks can config itself - so need to be careful
  • Perfmon, Debug & Trace
    • ties to CoreSight infrastructure (e.g. CTI)
    • timestamp will be brought from outside; not internally generated
      • there are ARM protocols for timestamping - e.g. compressed timestamp, and full timestamp coming into NoC, gray-coded or not
      • STPv2 on output of ATB uses compressed timestamp
      • Need more investigation on how this compressed / uncompressed timestamping will mean (in microarchitecture)
    • CoreSight component compliant necessity? Need careful thought due to configurability of probe
    • How about non-CoreSight users? Do we need a non-ATB output? (e.g. AXI4-Stream output)
45minResiliencyJohn Coddington (Deactivated)
  • resiliency can most likely be on a flow level
    • need to ensure packets from unprotected domains do not corrupt the protected domains, e.g. memory protection unit (MPU)
  • block and sub-block duplication
  • fail operational is the official nomenclature
  • end-to-end coverage (includes protection scheme from customer); will have overlap in the protection (i.e. by duplication)
    • in between the translation of the protection scheme, there could be logic that may not be covered if customers do not use duplication!
  • sub-block duplication: what is the cost? (delay logic, comparators, etc.). More investigation need to be done.
  • safety controller - will take in all possible faults (memory ECC, data ECC, etc.)
    • only BIST protected, not duplicated (ASIL D requires 90% latent fault)
  • ECC protection for memory - should also be reported without resliency / safety controller - we will have interrupts separate from the fault controller
  • signals going to safety controller may be physically far apart
    • will need to provide pipeline stages for these signals / async crossings (level signals) / ? to be investigated
  • fail operational
    • current POR in engineering:
      • stay safe for as long as you, and if failure, signal as fast as you can, and transfer over
      • in functional soc: add trusted check points and fail over
      • in fail over soc: how to resume as soon as possible
    • how about isolating errors? (e.g. not let errors propagate everywhere; reset the units that are corrupted)
      • e.g. timeout recovery (both initiator and target sides)
    • Charlie's suggestion: Ty Garibay (Deactivated) to discuss fail-operational with customers
