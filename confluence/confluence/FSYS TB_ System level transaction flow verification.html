Current Status

How to Enable

  • Add +EN_FSYS_SCB plusargs in testlists or add it to the runsim command. (-so ‚Äú+EN_FSYS_SCB=1‚Äù)

  • Which tests to enable for:

    • Scoreboard can be enabled for all FSYS tests. Error scenarios aren‚Äôt supported.

What is supported

  • CHI: all transactions.

  • ACE/AXI: all transactions.

  • FSYS regression tests with FSYS scoreboard enabled: All fsys tests. Except for error scenario tests.

  • Unique ID(FSYS_UID) assigned to transactions generated from AXI/CHI bus. Grep for FSYS_UID:<txn#> to see transaction‚Äôs progress through NCORE.

  • In case of fail, FSYS_TXN_Q gets printed which shows all pending transactions' details. Look for messages with tag [FSYS_TXN_Q] near the end of logfile.

Not Yet Implemented/ WIP

  • Memory consistency checks: In progress. Working on plan, schedule and resource allocation.

  • DMI AXI traffic: Current transaction tracing is upto input SMI ports of DMI. DII‚Äôs AXI transactions are monitored.

  • Known Issues:

    • FSYS_SCB prints an UVM_WARNING when there are multiple IOAIU CMDReq matches. The scoreboard testing will continue to make progress by matching incoming CmdReqs to the oldest AXI txn in age order.

    • RBReq from DCE can also have multiple matches with pending txns & there is no single distinguishing variable. Currently scoreboard matches the outgoing RBRs to pending txns in the order in which DCE saw CMDReqs, In the event of multiple matches, earliest CMDReq‚Äôs txn will be matched.

      • Open: RBR&MRD uses different SMI port than STR & SNP. STR&SNP use same SMI port. Sometimes STR is seen first by SCB and then RBR/MRD Or a SNP first and then RBR. Verif gets difficult when it can't rely on this order.

    • Proxy cache model is not yet implemented. Scoreboard uses well defined assumptions to predict IOAIU behavior when proxy cache is enabled.

    • Error scenarios aren‚Äôt yet supported.

  • Opens:

    • Look at list of open JIRAs below.

Goal

Details

1

End to end transaction tracking scoreboard

  • Implement transaction tracer. Transaction fields, details etc checking will/should be done at unit level.

  • Scoreboard will determine every transaction‚Äôs path through NCore. This transaction will be assigned a unique FSYS_TB_ID number and added to a pending queue.

  • At every unit‚Äôs interface, incoming/outgoing packets will be compared against pending transactions.

  • Transaction will be removed from the pending queue when it ends.

  • At the end of the test, scoreboard pending transactions queue should be empty.

  • For debug purpose, the scoreboard will print meaningful messages while transaction is making progress through NCore units.

  • Latency and bandwidth calculations.

2

Memory consistency checks

Memory Consistency Check‚Ä¶

Memory Consistency Check‚Ä¶

The memory consistency checker is implemented using heuristic technique. The checker uses forward looking and backward looking queues of transactions to evaluate the consistency. Given nature of multicore system with multi layer private caches and memory optimizations, the heuristic approach is used in this checker. Refer to above slide decks.

Use ‚Äú+EN_MEM_CHECK‚Äù simulation time option to enable these checks.
