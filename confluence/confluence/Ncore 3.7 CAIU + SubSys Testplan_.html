Ncore 3.7 Features

Interface Parity for CHI-B

  • Create CHI-B config with interface parity and resiliency enabled (enable async adapter in subsys/fsys config to check special handling of flitpend signal).

  • Support only ODD parity check on all signals (including sysco).

  • Note: This feature was implemented for CHI-E in Ncore3.6. Any new tests/check added CHI-B will be reflected for CHI-E as well.

Interface Parity with no error

Check that when TB drive the right parity on all the check signals, all the commands complete as expected and no error is logged/UC_IRQ/mission fault asserted.

Ensure the output check signals from the DUT signals are driven with correct parity.

For FLIT signals the check is done only when FLITV is high and for all remaining signal the check is done every cycle after coming out of RESET.

Interface parity with error

Check that when TB drive parity error on the check signal the errors are logged in the register and trigger UC_IRQ. The error logging and UC_IRQ should be according to IntfCheckErrDet and IntfCheckErrIntEn registers settings. Mission fault is always asserted when resiliency enabled.

What happens when the DUT output parity check fails???

Snoop commands

Generate various types of snoops (stash, regular non-stash, DVM snoops) to the DUT. Throttle the snoops such that STT entries become full and not full multiple times.

Regular commands are issued from the CHI interface to occupy OTT entries.

All request flit and snoop fields valid values are randomized.

#Stimulus.CHI.v3.7.MaxSttEntries

High

Not Started

If Skid buffer is implemented STT entries might be smaller so have to check total outstanding snoops. Revisit after design is finalized.
