Notes:

Please find the minutes of the CodaCache PRD re-discuss. 

Status of the re-discussed items:

  • Limitation of the transfer size to 64 Bytes (today position is that an upstream FlexNoC would chop the traffic down to 64B for long bursts)

Status: recommendation to drop this restriction and accept any kind of transfer that and AXI Interface can provide.

SMART: decision to release 1.0 with 64B restriction. Full AXI support deferred to 1.5. Question sent to Sumie

  • Cache allocation overwritten with CSR registers (today not supported)

Status: recommendation to allow csr registers to overwrite the allocation based on transaction attributes  

SMART: implemented in 1.0

  • Address range flush support (is no from 1.0 PRD but is implemented as yes à approve)

Status: Agreed. PRD to be updated accordingly

  • Read response interleaving support (should be re-discussed, is no today)

Status: Recommendation to drop this restriction. Read Data interleaving must be supported in 1.0

SMART: decision to release 1.0 with 64B restriction. Full AXI support deferred to 1.5. Question sent to Sumie

  • Partial write allocation (in case of miss need a prefetch logic, Khaleel liked to eventually move this feature to later releases)

Status: Agreed to defer the prefetch logic for partial write miss to later release if engineering feels it necessary

  • RTT and WTT sizes limitation to 64positions (is up to 128 in PRD, need to approve the change)

Status: Agreed. PRD to be updated accordingly

  • Memory sleep support (need to be re-discussed, this is very technology dependent)

Status: Agreed to defer memory sleep support to later release. This item will need further discussion, we could drop it from the program

  • End to End protection. This was a Mobileye requirement but not fully documented. The CodaCache has been dropped by Mobileye since. I am in favor of a very simple design instead like place holder in the AXI interfaces (no more).

Status: This end to end implementation is dropped from the program. If a new request comes in, we will follow the process: MRD, PRD ….

  • Assess the need of a TLM model for the CodaCache 1.0

Status: The TLM model is a MUST for 1.0 release. We must create a MRD, PRD and reflect it in our Roadmap

SMART: Priority order is first CMC/Proxy, second CodaCache.

  • Qchannel

SMART: Engineering to come back with a position for 1.0.

Qchannel is highly desirable for CodaCache shutdown.
