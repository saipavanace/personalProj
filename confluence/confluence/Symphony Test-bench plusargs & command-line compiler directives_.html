Symphony Test-bench plusargs

Using +seq run-time plusarg :

Structure of +seq run-time plusarg :

 +seq=Phase-Order-Sequence_name-Initiator_number>,+seq=<Phase-Order-Sequence_name-Initiator_number>….,+seq=<Phase-Order-Sequence_name-Initiator_number>


Here,

Phase = Name of phase in which stimulus is generated. Mostly we use main phase.

Order = Order of traffic generation on that particular interface.

Sequence_name = Give your sequence name here.

Initiator_number = Specify  number of initiator interface.


Below are some variations of +seq run-time plusargs and see its effect on stimulus. When +seq is being used, It becomes directed test, then user has to be specific about Phase, Order of intended sequence, Sequence name, and Initiator number.

Run-command:

runsim -e top_axi_axi -t axi4_multi_beat_wr -c maestro_4x4_AXI4_64_256_256_256_32_AXI4_32_32_32_256_256 -w -n 5 -so <runtime_argument>


Traffic in a sequential manner :

A) runtime_argument= “+seq=main-0-axi4_sequence_0-0,+seq=main-1-axi4_sequence_1-1,+seq=main-2-axi4_sequence_2-2”

After applying above run-time argument, you can see that axi4_sequences are generating traffic in order according to order in argument.

First Sequence - axi4_sequence_0 generates traffic on Initiator-0 until sequence finishes as it has been given order - 0. Next comes sequence - axi4_sequence_1 to generate traffic on Initiator-1 as it has been given order - 1 and then sequence - axi4_sequence_2 to generate traffic on Initiator-2 as it has been given order - 2. Initiator-3 does not have any traffic during the simulation since It is not mentioned explicitly using +seq.

So, In this case, axi4_sequence_0, axi4_sequence_1, axi4_sequence_2 would generate stimulus one after another on Initiator-0, Initiator-1, Initiator-2 interfaces in sequential manner. No traffic on Initiator-3 would be generated.


B) runtime_argument= “+seq=main-2-axi4_sequence_0-0,+seq=main-0-axi4_sequence_1-1,+seq=main-1-axi4_sequence_2-2”

Here, Looking at the order, axi4_sequence_1, axi4_sequence_2, axi4_sequence_0 would generate stimulus on Initiator-1, Initiator-2, Initiator-0 interfaces in a sequential manner. No traffic on Initiator-3 would be generated.


C) runtime_argument=“+seq=main-1-axi4_sequence_0-0,+seq=main-2-axi4_sequence_1-1,+seq=main-0-axi4_sequence_2-2”

Here, Looking at the order, axi4_sequence_2, axi4_sequence_0, axi4_sequence_1 would generate stimulus one after another on Initiator-2, Initiator-0, Initiator-1 interfaces in a sequential manner. No traffic on Initiator-3 would be generated.


Concurrent traffic :

runtime_argument = “+seq=main-0-axi4_sequence_0-0,+seq=main-0-axi4_sequence_1-1,+seq=main-0-       axi4_sequence_2-2”

runtime_argument = “+seq=main-1-axi4_sequence_0-0,+seq=main-1-axi4_sequence_1-1,+seq=main-1-       axi4_sequence_2-2”

runtime_argument = “+seq=main-2-axi4_sequence_0-0,+seq=main-2-axi4_sequence_1-1,+seq=main-2-       axi4_sequence_2-2”

Here, you can see that order of all axi sequences are same. In case order is same (any number 0, 1, or 2), all axi sequences would generate traffic on its respective initiator in parallel.


Don’t use cases :

A) runtime_argument= "+seq=main-0-axi4_sequence_0-0,+seq=main-1-axi4_sequence_0-1, +seq=main-2-axi4_sequence_0-2"

Setting AXI4 VIP delays via +ARGS:

Master VIP delays:
// ------ AW delays ------ //
rand int awdelay; int awdelay_min; int awdelay_max;
// ------ AR delays ------ //
rand int ardelay; int ardelay_min; int ardelay_max;
// ------ W delays ------ //
rand int wdelay[]; int wdelay_min; int wdelay_max;
// ------ R delays ------ //
rand int r_ready_delay; int r_ready_delay_min; int r_ready_delay_max;
// ------ B delays ------ //
rand int b_ready_delay; int b_ready_delay_min; int b_ready_delay_max;


Slave VIP delays:
// ------ AW delays ------ //
rand int aw_ready_delay; int aw_ready_delay_min; int aw_ready_delay_max;
// ------ W delays ------ //
rand int w_ready_delay; int w_ready_delay_min; int w_ready_delay_max;
// ------ AR delays ------ //
rand int ar_ready_delay; int ar_ready_delay_min; int ar_ready_delay_max;
// ------ R delays ------ //
rand int rdelay[]; int rdelay_min; int rdelay_max;
// ------ B delays ------ //
rand int bdelay; int bdelay_min; int bdelay_max;


All min and max delays can be set from plusargs

The AXI4 driver can set : aw_ready_delay, w_ready_delay, ar_ready_delay, r_ready_delay, b_ready_delay, bdelay, rdelay

The master VIP axi4_seq_item can set : awdelay, ardelay, wdelay

The slave VIP axi4_seq_item can set : aw_ready_delay, w_ready_delay, ar_ready_delay
To give delays from slave sequence the plusarg "axi4_slave_delay_seq_item_control" has to be passed otherwise delays are randomized in the driver

Eg for plusarg:-
runsim -e top_axi_axi -c maestro_4x4_axi_axi_exclusive_coverage_20jul20 -t axi4_single_beat_uniq_id_across_init_exc -w -so "+ar_ready_delay_min=0,+ar_ready_delay_max=30,+axi4_slave_delay_seq_item_control=1"


Constraint explanation for channel delay:-
IF   
value of max_delay and min delay differance > 5 then :       xdelay is distributed in the following conditions-
[min_delay : min_delay+ roundoff value of ((max_delay-min_delay)/10)] :/ 10 (Distribute in 10 priority) 
[min_delay + roundoff value of ((max_delay-min_delay)/10)+1 : max_delay - roundoff value of ((max_delay-min_delay)/10)] :/80 (Distribute in 80 priority)
[max_delay - roundoff value of ((max_delay-min_delay)/10)+1 : max_delay ] :/ 10 (Distribute in 10 priority)

ELSE
min_delay <= xdelay ; xdelay <= max_delay

Here xdelay = awdelay , wdelay , bdelay , ardelay , rdelay, aw_ready_delay , w_ready_delay , b_ready_delay , ar_ready_delay , r_ready_delay

For roundoff:
If roundoff value for real number >= 0.5 it takes ceil value
If roundoff value for real number < 0.5 it takes floor value
