Command to open the tool:

vsim -viewcov UCDB_FILE -do EXCLUSION_FIFE

See uncovered code:

Click on the instance to view the rtl code. XE, XS, XC shown per line stands for uncovered Expression, Statement and Condition. See Coverage Detail below for more information.

Coverage Analysis:

For a more detailed analysis and summary, open the analysis view, you can show the wanted coverage type at the upper right cover of the window (condition/statement/.., covered/uncovered)

Steps: View → Coverage → Code Coverage Analysis

Coverage By Instance:

Default is showing coverage per module.

Steps: Tools → Code Coverage → Show coverage by instance

Recursive Coverage View:

With recursive coverage view disabled, the % bar shows coverage as local view (only the instance). You can set the recursive coverage to see if there’s any holes in all the sub instance:

Steps: Structure -> Code Coverage -> Click on Enable Recursive Coverage

Coverage Details:

Open the coverage detail window to show hit/miss details.

Steps: View → Coverage → Details

Coverage Report:

Sometimes coverage details is not able to show all coverage holes (Mentor AE said it’s a bug), you can still identify them in text report.

Steps: Tools → Coverage Report → Click on Text


You can export the exclusion to a .do file or append to a current file with the option checked.

Steps: Tools → Coverage Report → Exclusions

Coverage Metrics display options on GUI:

In case you are not able to see certain coverage metrics on the GUI. Follow to the below steps to control the coverage metrics displayed on the GUI window.

1. In the *_merged (Recursive Coverage Aggregation) - Default Tab. Right click anywhere on the title where all the coverage metric names are mentioned. (In the below screen shot right click next to FEC Expression graph)

2. Select "Change Column Visibility" option

 

3. Check required metrics to be displayed and click OK.

Questa FSM Coverage Limitations and Coding Guidelines.

Unsupported FSM Design Styles

Questa SIM does not recognize the following design styles:

  • FSMs using complex “if” conditions where a current state variable is ANDed with another current state variable.
  • Using Verilog part-select expressions as a current-state variable.
  • Using VHDL slice expressions as a current-state variable.
  • Defining a single FSM in multiple modules.
