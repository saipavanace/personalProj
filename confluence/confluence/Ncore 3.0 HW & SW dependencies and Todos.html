HW SW dependencies 

 

HW generation flow dependencies

  • CPR tool capabilities 
    • Capability to define parameters for all Ncore and legato / presto units and reference other sub units.
    • Capability to define parameters for system level and create dependencies
      • Capability for a unit to get parameters calculated based on other units within the system & global parameters
      • System level derivations and checkers 
    • Capability to define interfaces and reference them from other units
    • Capability to define CSRs and dependencies between CSRs and other parameters of the system 
      • Capability to reuse CSR i.e. global CSR definitions   
    • Capability to support export generation for customer and  internal documentation
      • CSR information 
      • GUI Parameter information and descriptions
      • GUI labels and tool tips.   
  • TACHL Support
    • Interfaces
      • dynamic derivation of interfaces of modules, this refers to TACHL syntax and capability for RTL designers to specify them. (need a sperate meeting to capture all requirements)
    • Miscellaneous open JIRAS
  • Verification support 
    • verification TB generation support (support to generate SV interface construct)
      • Generation of top level RTL interface & test bench interfaces.
      • Auto stitching of above virtual interfaces to physical RTL wire I/O interfaces.
    • Data structure input to verification environment so that it understands what the system looks like. (system JSON) 
  • Generating Jsons 
    • capability to generate unit level jsons from parameters specified in CPR. This includes the capability to validate parameter values 
      • Use all default values
      • Use an input file which specifies some parameters
      • Randomize the parameters within legal valid limits.
      • Randomize the parameters within user specified constraints  
  • Generating system level i.e. top level RTL that instantiates different Ncore and legato components (Topology aspect to be discussed)
    • Initial version could be simple with fixed topology and parameters
    • Second version can have added capability to specify some parameters that do not change topology, this could be passed in using a txt file
    • Capability to randomize parameters that do not change topology.
    • Capability to create multiple specified topologies
    • Capability to randomize topologies
