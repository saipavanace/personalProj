1.Introduction


This document provides a detailed testplan for verifying DCE. The initial sections describe the test bench high level architecture. The functionality of the main components and how they interact with other components are discussed. The later part of the document discusses DCE configurations that are part of the nightly regression. All these APF’s are configured so that all NCOREv2.0 features with respect to DCE are covered. At the end, the document goes over all the tests executed.

2.1 TB diagram

2.2 List and Description of the TB components

Below are the list of components DCE unit test-bench instantiates.

            Test-bench:  module dce_tb_top ($CONCERTO_TOP/dv/dve/tb/dce_tb_top.sv) is the top module that instantiates DUT to respective interfaces. Instances of the interfaces are forwarded to test-bench using uvm_resource_db methodology. This module also probes certain RTL signals that are needed for Checkers/Coverage collectors and assertions. At time 0, creation of all test-bench components are initiated by calling run_test() method.

            dce_unit_test: This is the high-level class ($CONCERTO_TOP/dv/tests/dce_unit_test.sv) that creates all the UVM components from top to bottom. Depending on the knobs (plusargs), respective Sequences are constructed and stimulus is driven/collected. This component constructs DCE environment and DCE environment configuration object. Various tests and are described latter in this document.

            dce_env: This component will build DCE environment by instantiating all agents and checkers. ($CONCERTO_TOP/dv/dce/env/dce_env.svh). SFI Master/Slave agents OCP agent and IRQ agent are instantiated. There are two checkers in DCE environment, directory manager Cycle accurate checker and DCE checker to check all other logic expect for Directory manager lookups. Both the checkers are ($CONCERTO_TOP/dv/dce/env/dce_dirm_scoreboard.svh) and ($CONCERTO_TOP/dv/dce/env/dce_scoreboard.svh). In addition, dirm checker instantiates directory model (behavior model to mimic RTL behavior) for stores and checks. More detailed explanation on various checks to follow on.

4. Checks

 

DCE environment has 2 checkers.

  1. Directory Manager Checker 
  2. DCE Checker

    The Directory manager checker functionality is to correctly predict/check Lookups and the ATT linked list order. 
    The DCE checker checks all other checks expect for Directory manager lookups. These values are inputs to this checker that is probed from RTL.

4.1 Directory Manager Checker

 This checker implements a behavior directory model that mimics the behavior of RTL.

Checks

Description

ATT linked list check

Predicts which transactions after P2 will be Active or go to sleep. Also, checks that transactions wake up in correct order.

Lookup request check

Records the transaction information on lookups and then checks that given {olv, slv} values are correct on P2 (directory response). Predicts olv, slv values by lookup into directory model.

Update request check

Determines if the Update request is going to be dropped or if it clears the valid bits of olv/slv.  On next lookup to same cacheline, checks that prior update has taken right effect.

Wake request check

Checks if transactions are expected on Wake interface. Also, checks that given {olv, slv} values are correct on P2. Prior committed values are read.

Recall request check

Predicts accurately when and which cache-line is expected to be Recalled. Also, checks that given {olv, slv} values are correct on P2.

Maintenance recall request check

The model mimics the behavior of Maint-op scheduler in CSR logic. Once the CSR’s are programmed, cycle accurate check is implemented to predict value of MaintActv signal.  Checks that Maint operation address or index/way are correct when Directory maint look up happens. Predicts and checks if Maint request will be dropped or if request will forward progress. Checks that prior olv, slv values are correct.

 

For recall all and recall victim buffer, checker is cycle accurate in predicting which entry will be recalled. Compares the cacheline address index and way.

Uncorrectable Errors

When uncorrectable error happens, olv, slv values are accurately predicted for P2 responses for this request and all other requests.

CSR access to Directory registers

Checker monitors the traffic on CSR interface and understands the various Maintenance operations and SF enables/disables. Understands memory initializations operations and performs similar operations in behavior model so that model is cycle accurate with RTL


4.2 DCE Checker

This is a high level checker for entire DCE end to end operations. This checker reads the traffic on SFI Slave request channel and predicts the outcomes on SFI Slave response channel, SFI master request/response channel. Expect OLV, SLV values, this checker predicts all other outcomes. The OLV, SLV values are read directly by probing RTL. These checks are performed by the Directory manager scoreboard.

Also, checker receives certain RTL internal status information on a separate interface to accurately predict on power management sequences and in-correctable errors injections. This interface is required to have precise checks on these events. This interface is also required for additional checks to RTL internal status signals.

Various checks performed by DCE checker are mentioned in below table

Checks

Description

Command response checks

Checks that for a transaction command response is received before the transaction is deleted from the checker.

 

For DVM requests, checks that only one command request is pending for a given DVM capable agent at any given point of time.

Update response checks

Checks that for a transaction update response is received before the transaction is deleted from the checker

Snoop request checks

On Directory response P2, predicts which caching agents will be snooped and compares the entire SFI packet received from RTL with predicted packet.

If ever agents are powered down before these snoop requests are issued, then by probing RTL signal, model predicts which Snoop requests are deleted.

Hint request checks

Checker predicts if Hints are expected for a given transaction. Also, checks all the fields in received SFI packet.

MRD request checks

Checker predicts if MRD’s are expected for given transaction after Snoop responses are received or predicts in P2 itself if Snoop requests are not expected.  Also, checks all the fields in received SFI packet.

STR request checks

Checker predicts STR request expected and compares with RTL packer received. For all the above requests, all SFI fields are compared expect for SFI pressure, SFI transId and resiliency related field if any.

DTR request checks

Checker predicts if DTR Error request is expected from DCE.

Snoop response checks

Checker checks if all Snoop responses are received before issuing either MRD or STR or commit request happens. Also, triggers error if a response received from an agent is illegal for issued request.

MRD response checks

Check is similar to that of above explanation

STR response checks

Check is similar to that of above explanation

DTR response checks

Check is similar to that of above explanation. Additional check is that only one DTR response is pending at any given point of time.

Directory commit checks

After StrRsp is received (for recall requests, it’s after all snoop rsp’s are received) Commit vector ocv, scv is predicted by the checker. When Directory manager receives transaction on commit response interface, RTL’s ocv, scv vectors are compared with predicted values. This provides end to end check of Directory checker.

DVM Snoop/STR checks

For DVM transactions, all coherent checks are valid. The additional check is that the depth DRB & DSB fifo’s are correct. Meaning only one active transaction is pending for DVM snoops. Similarly, only one DVM transaction is waiting for STRrsp in DSB fifo.

ace_exokay bit check

Checker predicts which exclusive store operations will be granted access and which ones will be denied. This bit is checked with respective RTL bit value when StrReq show up on SFI master request channel. Checker has the knowledge of Tagged monitors in the system and implements behavioral model to predict the value. Also, checks that Snoop Requests are not issued for failed exclusive stores and STR request is issued right away.

transActive check

This is cycle accurate check. Whenever RTL value of this signal is low, Checker checks if this is true. Check is disabled if this value is asserted

snpActive check

Similar to above expect that for each caching agent this check is done individually

dvmActive check

For all DVM snoop capable agents, checks are performed.

Snoop Credit checks

Checker implements a behavioral model to check that for a given caching agent, number of outstanding Snoop requests never crosses maximum number of credits available.

MRD credit checks

Similar check for MRD requests.

SFI Trans-Id check

If ever there are two active transaction with same sfiTrans-Id then checker triggers error.

Deletion Check

Before de-allocating the ATT-id all packets received check is performed.

 

4.3 Miscellaneous Checks

 

Checks

Description

SFI Pressure checks

These cycle accurate checks are in dce_tb_top. Predicts the expected SFI pressure value and compares them with RTL SFI master request pressure value. Also, checks that RTL counters do not overflow or underflow

Interrupt trigger logic

Whenever Correctible or uncorrectable interrupt is asserted, CSR access is performed from sequences the read the contents. On uncorrectable Recall errors, contents are compared with values latched while injecting error. For other scenarios, we simply clear the interrupt.

 

When errors detection or correction in disabled, test failure is triggered if interrupts are asserted.

Snoop credit empty check

At the end of simulation, dce_tb_top checks that there are no pending snoop credits

MRD credit empty check

Similar check for MRD credits

ATT-ids used count

At the end of simulation, the are no ATT-ID pending

Error CSR’s are 0’s

If no errors are injected then endo of sim check for Error CSR values.
