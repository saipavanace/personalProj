Options for code coverage confidence

Modified Traditional Coverage Analysis

The first step was done in 3.2.0. The downside is that it may take some time to achieve parity with 3.2.0 code coverage results.

  1. Get code coverage results up to 99%, 95%, 95% by running coverage regressions, updating formal and RTL exclusions and writing new stimulus.

  2. RTL examines all remaining holes, and finds only those that relate to new/changed RTL.

  3. The holes found in step #2 are addressed through stimulus and waivers.

Delta Coverage Analysis

The downside of this plan is the very manual effort done by the RTL team to analyze RTL changes and coverage in steps 2-4.

  1. Get code coverage results by running coverage regressions, updating formal and RTL exclusions and writing new stimulus. Do not iterate to get 99%, 95% and 95%.

  2. RTL lists all lines of IOAIU “tachl RTL” change have changed since 3.2.0 release.

  3. For each of the seven coverage configs, figure out which lines of “generated RTL” changed based on the “tachl RTL” changes. There are seven configs and about 120-140 lines of tachl RTL changes. If there is a 1 for 1, then there 1000 lines of changed generated RTL. Note that design will need to say how much of the IOAIU RTL is tachl.

  4. Design goes through the coverage for the seven configs, having the list of generated RTL changes, and goes through each of seven coverage html reports. Design will create a list of changed RTL holes to discuss with DV.

  5. The holes found in step #4 are addressed through stimulus and waivers.

Note that RTL waivers may still be needed for new RTL, because the IOAIU RTL is not fully tachlized.
