Terminologies

 

Requirements:

  • ASIL-D ready
    • Unit or path duplication
      • Requires comparison logic, and checker for comparison logic
      • Must not compromise performance with duplication
      • Comparison logic must be protected
      • Functional and checker units must run on different clock nets (so do comparison logic and safety controller - Monica to gather more details)
      • Duplicated unit must be run in lock-step with the functional unit - the duplicated unit must be delayed by 1 to N cycles, configurable in design time (Monica to confirm)
    • Parity or ECC protection (header and payload)
      • Ability to connect IP with different protection schemes  to the interconnect (i.e. IPs with parity support, ECC support, or no protection at all)
        • These IP may have only protection on data, or protection on both data and address (Monica to double check)
        • There should be minimal RTL code that the customer needs to write (e.g. for more "homogeneous" systems, there should be no need for RTL coding on the customer side)
      • It is possible that all IPs do not have protection scheme, but protection scheme is required internal to the interconnect
      • For storage applications: there is need for adding parity/ECC on top of parity/ECC that the IPs are transporting across the interconnect
    • ARM R5, R7 interface (AXI port error detection) support
    • ARM R52 (Kyte) interface support
    • Ability to detect cases such as:
      • Deletion of a valid packet or creation of a bogus packet (e.g. bit flip on the valid signal for single beat packets)
      • Length of packet not matching packet header (e.g. bit flip on the valid signal for multi-beat packets)
      • Packet reaching wrong destination (route ID bit flip)
      • Packet stuck at a router
      • Handshake protocol consistency? (e.g. illegal sequences of ready/valid handshake such as retraction)
      • Control register bit flip causing functional error
    • Safety controller
      • Mission fault and Latent fault maskable interrupt
      • allow software to identify the source of failure
  • Fail-operational
    • In addition to detecting a fault, the interconnect should have the ability to safely recover from the faults
      • Monica to further collect requirements on the types of faults that need to be covered
    • In case of unit duplication, a method to run diagnostics on both the functional and checker units (can be scan, or bist, etc.).  Upon detection of fault, a means to switch to using the remaining oprational unit.
  • Mix of ASIL-B and ASIL-D domains in a system and ensuring there is no contamination or interference
    • An example is Memory protection unit (MPU) - ARM
  • Symphony software must be flexible to allow users the granularity of what resilience features are enabled, and what units are duplicated
  • Safety documents (partner with 3rd party expert)
  • Safety process during Symphony development
