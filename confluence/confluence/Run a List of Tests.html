Purpose and Philosophy

The primary intent of this tool is to run a set of tests with randomization on the queue. To keep it flexible, it was designed to work with scenario files, so that that all the details of running can be captured in scenario files only the minimum amount of information is required to be supported by the tool. The tool understands

  • what a scenario file is, what the format is, expecting it to be named with .scn extension
  • what a json file is, expecting it to appear in scenario file with maestro-config line
  • what a random seed is
  • what an input file is, and expecting it to appear in scenario file with a path that may need to be adjusted
  • all other parts of a scenario file are blindly copied to new random seed tests, without being parsed/adjusted
  • any additional command line options to run the executable can be passed in quotes 
  • the summary will include number of fails by signature type, as long as the error in log includes a keyword, or starts with [Error]

Queue Usage

If you run run_list.pl with the --queue option, it will submit jobs to the grid and wait for them to complete. During that time, it will display the command to kill the jobs on the queue in case you ran the command by mistake or you feel there is a job that is taking too long. It is expected that current simulations should generally take 5-10 minutes or 30 minutes maximum if they reach the SystemC timeout. If they take longer than 2 hours, they may automatically be killed by the grid itself. You can see your queue jobs at any time by doing

qstat

You can delete all your jobs on the queue by doing

qdel -u <username>

Debugging fails

  1. Look at report.txt to find failing signatures and number of fails for each signature
  2. grep for the failing signature in *.log to find which file contains the signature
  3. the scn to reproduce the fail is the scn file with the same base name as the log file.
