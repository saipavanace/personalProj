6.ATUI_AXI

Errors:

Internally Generated Errors:

  Buffered Write: 

      1. Create TCL config with enBufWrite = True i.e., enabel buffered write.

      2. Update TB slave drivers (AXI4/APB3) to send slave error.
             AXI4 slave: bresp = 2
             APB3 slave: PSLVERR = 1

      3. Enable interrupts at buffered write registers through RAL.

      4. Send regular AXI initiator traffic.

      5. Expect Interrupt due to error, Interrupt service routine should automatically clear the interrupt. Do we need to do anything special?

  RW Error: 

      Not supported for R1, because writeSupported and readSupported are network parameters and if they are set to false initiators won't be able to send corresponding transactions.
      Note: If Maestro sets up this parameter incorrectly then we should have a maestro checker to catch this issue.


  Timeout Error: 

  1. Create TCL configurations with ATUI socket parameters set to values provided in below table. 
  2. Increase TB slave driver response time > timeoutErrCount no. of cycles.
  3. Enable interrupts register values for timeout through RAL.
  4. Send regular AXI initiator traffic.
  5. Expect Interrupt due to error, Interrupt service routine should automatically clear the interrupt. Do we need to do anything special?

Note: For a given transaction once error is asserted, cleared, it will not be asserted again even if slave doesn't respond at all. Do we need a check for this?

ATUI parameterValues
timeoutErrChkthis will be set when timeoutErrCount> 0. No need special set-up
timeoutErrCount100 (Some small value), represents no. of cycles
timeoutUseExternalValue

Sync-up with Junie regarding timeoutUseExternalValue set-up


  Decode Error:

  1. Create TCL config with enDecodeError = True for ATUI in rtl.json.
  2. Enable interrupt registers for decode error through RAL.
  3. Send transactions with address not present in pam table. i.e., PAM table should be complete without default and send transaction with address not present in PAM table.
  4. Expect Interrupt due to error, Interrupt service routine should automatically clear the interrupt. 

  Exclusive monitor error:

  1. Create TCL config with: Splitting enabled
                                          Some of the Targets: no. of exclusive entries > 0 and wrapSupported = false
                                          Initiators: wrapSupported = true
  2. Enable interrupt registers for Exclusive monitor error through RAL.
  3. Send exclusive wrap transactions to the target with no. of exclusive entries > 0 and wrapSupported = false and expect exclusive monitor error.
  4. Expect Interrupt due to error, Interrupt service routine should automatically clear the interrupt. Do we need to do anything special?
  5. Do we need to test every splitting condition?

  Power domain error:

  1. set parameter pamTargDomain (domain each transaction has to go through) = Each entry is an object and there is one array of integers. (Needs to be set-up by maestro).
  2. Signal domain interface = array of bits indicate domain is on or off. reset this parameter.
     Eg: Set pam target domain to array of 4,5,7
          Set domain interface bit 4 or 5 or 7 to 0 to get the error.
    Note: Neither of these parameters are currently present in Meastro generated rtl.json and needs implementation. Expected to be implemented for R1 silver.
  3. RamaKrishna Vyata (Deactivated) to work with Maestro and design for more set-up information.

For no. of interrupt accumulators present:

  1. Perform any of the set-up for ATUI_AXI, ATUI_APB, ATUT_AXI and ATUT_APB above.
  2. Enable/Disable different levels of interrupt accumulators using RAL programming.
  3. Predict Interrupt generation.
  4. Clear Interrupts at lower level blocks but not at the accumulator, then interrupt should stay. Needs ISR modification.
  1. check if interrupt generation happens as predicted
  2. Interrupt service routine functions properly. Ex: Interrupt timeout error?
  3. If there is outstanding interrupt at end of sim, trigger an error.



11.Cross with any other features?

  1. Splitting - every type of error.
  2. buffered writes - every type of error. In configs for error generation have buffered write enabled.
  3. Non-modifiable Narrow enabled with every type of error.
  4. re-order buffer with timeout error.




12.Set-up to try and break the design

  1. Enable Error generation set-up in RAL but don't enable interrupt generation.
  2. Enable interrupts but don't set-up for error generation.
  3. Will we be set to fail if we get unexpected interrupts? And we have interrupts set-up for most of the design.
  4. Set-up more than 2 interrupts in a given config and see if it causes any issue.
  5. Set-up with back to back interrupts, how will TB and design handle it.
  6. Set-up multiple overlapping interrupts and how would system handle it.
  1. TB should not observe interrupt, flag TB checker if Interrupt is observed when not predicted.
