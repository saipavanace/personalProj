• It is LEGAL to use QtCreator as a development environment to edit Maestro files and to debug its execution.

  • You need a Qt license to update and commit GUI files that use Qt APIs. Unless you are one of the Maestro GUI developers, abstain from that.

 

The following instructions assume a clean clone of Maestro.

 

ssh -X user@gr6-3.arteris.com

module load maestro

cd /path/to/maestro

[Assumes a clean checkout, but may work anyway; you can try "make clobber"]

make cmake_debug cmake

qtcreator&

[If asked with a popup to update QtCreator, refuse]

File->Open File or Project -> choose the toplevel CMakeLists.txt in the maestro folder

[Some parsing should go on in the log window]

[If asked about which kit to use, pick one. Enable the Debug and Release checkboxes]

In the left QtCreator pane, Projects -> Build Settings

  • You can remove any build configuration other than Debug and Release
  • For the Debug configuration: choose as Build directory /path/to/maestro/build_debug
    • This ensures that files are built in the same directory either when compiling from the command line or from QtCreator, which saves time and confusion
  • "Apply Configuration Changes"
  • For the Release configuration: choose as Build directory /path/to/maestro/build
    • This ensures that files are built in the same directory either when compiling from the command line or from QtCreator, which saves time and confusion
  • "Apply Configuration Changes"

In the left QtCreator pane, Projects -> Run Settings

  • In the Command Line Arguments textbox, you can choose "-c" to launch Maestro in console mode, or other switches, e.g. to load a script automatically

In the left QtCreator pane, by clicking on the icon that looks like a monitor, you can now choose:

  • Whether to build Maestro for Debug or Release
  • What to actually run, e.g. "maestro" (the main Maestro executable) or "libdata.utests" (in general, any of the Maestro unit tests)
In the left QtCreator pane, by clicking on either the "Play" icon (Run) or the "Play with bug" icon (Start debugging), you can now launch a compilation and execution of whatever was just chosen.
You can monitor the compilation in the "Compile output" window (Alt-4).
You can inspect compilation issues in the "Issues" window (Alt-1).
You can monitor the execution output in the "Application output" window (Alt-3).
If the Debug build was chosen and the "Start debugging" button was used, any failure (e.g. crash) will now leave you in front of the call stack in gdb.
