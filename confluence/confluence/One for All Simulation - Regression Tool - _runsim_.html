**************Run Simulation/Regression Help Page***************

Usage: runsim [options] //One of the option -e (Environment), -c (Configuration), -t (testcase) or -l (label) is required to run simulation

Options:

-add_cfg <string>: Add New Cfg/s             - Specify the directory of gen_cfg to add in testlist dynamically
-c <string>      : Configuration             - Specify the configuration to run
-co <string>     : Compiler Options          - Specify the Compile Time Arguments for SV Compiler
-d <string>      : Dump Path                 - Specify the path to generate *debug* directory hierarchy at specified location instead of default $WORK_TOP
-e <string>      : Environment               - Specify the block TB to generate the simulation for
-exe <hex>       : Execute Processes         - Specify the hex of processes to execute. Exits once any said process fails. Default is 0x1F7.
-f <string>      : json File                 - Specify the test json to use
-g               : Grid                      - Specify this option to run on the "grid", by default runs locally
  -intq          : Interactive Queue         - Specify this option to run on the interactive queue on new"grid"
  -lq            : Long Queue                - Specify this option to run on the lonf queue on new "grid"
-i <n>           : Iter                      - Specify the no of iterations for simulation tests, Give -i 0 to run only compilation
-l <string>      : Label                     - Specify label to generate particular simulation/regression
-m <string>      : Mail                      - Specify email address to forward the report
-n <n>           : +NUM_TRANS                - Specify the number of transactions for simulation
-p <string>      : prep Options              - Specify json files for prep Options
-pof <hex>       : Proceed on Fail           - Specify the hex of processes to run regardless if the process before it failed. Default is 0x000.
-q <string>      : qrSim Options             - Specify any additional qrSim Option alongwith it's switch alphabet
-qc <n>          : +UVM_MAX_QUIT_COUNT       - Specify the max quit count. Default is 1
-r               : Re-Run                    - Specify this option to Re-run bucketized failures
-ri <n>          : Random Config Iter        - Specify the no of iterations for js_randomized config generation
-rm_cfg          : Remove Cfg/s-Tests        - Specify to remove -c configs from testlist & git repository dynamically. If -t testname is provided then only specified -c configs will be detached from it's -t testnames
-rs <n>          : Random Config Seed        - Specify the seed for js_randomized config generation
-rto <string>    : TACHL immOverride Options - Specify the -I immediateOverAll option of Tachl. The json formatting example is -rto {\\\"assertOn\\\":true}
                                               Please take note the quotes and the escape both need to be escaped, there will be 3 escapes per quote
-run <string>    : Modes (tachl/sim)         - Specify this option to run single processes like tachl codegen or simulation only
-s <n>           : Seed                      - Specify the seed for simulation tests, Will only run one simulation
-so <string>     : Simultor Options          - Specify the Run Time Arguments for Simulator
-t <string>      : Testname                  - Specify the testname to run
-tcl <string>    : runmaestro tcl_cfg        - Specify the file path of tcl_cfg for run_maestro
-tf <string>     : tACHL Filename            - Specify the tACHL file when using '-run tachl' mode for RTL CodeGen
-to <n>          : +UVM_TIMEOUT              - Specify the timeout time in ps. Default is 15000000000ps (15ms)
-v <string>      : +UVM_VERBOSITY            - Specify the verbosity for simulation(UVM_HIGH/MEDIUM/LOW) Default is UVM_NONE
-w               : Wave                      - Specify to enable waveform dump
-z <string>      : cpr parameters            - Specify the cpr parameters to be added to the genTachl. This option is appended with "z" param in pkgMap.json
-h               : Help                      - Display this static Help, For Additional help refer below section.

Example:
runsim -e top_ctl -l all -exe 1ff -pof 8     // Runs "all" tests for "top_ctl" Environment(-e), executing processes at bit positions 0-8 and proceeding even if failure on process at bit position 3 (RTL Verilog Compile).

As mentioned in the "Example" in help page, user can run any simulations by giving the "-t" testname (or -e environment / -l label) and any additional options if required.

  • The tool generates a report after each run. Consider the following example:

    runsim -t top_axi_atu -n 10                            ( -n 10 sets +NUM_TRANS=10)

  • runsim filters testcases from -e Environment / -c Config/ -t Testname / -l Label options, anyone of these options is always required to run a simulation.

  • The tool requires a testlist.json file to run relevant configurations. Refer section B) testlist.json for more information.

  • The tool parses this testlist and provides A) Custom Help with the testcases/environments/labels to determine before doing a run.

B) testlist.json:

Configure your specifications in the testlist.json file.
OR
Make your own json file maintaining the SAME format as the testlist.json and pass it in runsim as a "-f" option (runsim -f path/to/myTestlist.json -l all)

  1. Add your testcases inside "testlist" : { "yourTestCase" : { "blkTB": ..... } }.
  2. The options marked in LIGHT BLUE sitting at the 1st level of nest are the DEFAULT options. If any of the testcases inside testlist have missing fields, default options values will be considered there. For example - 

    "testlist" : {

    "switch_static1": { "blkTb": "switch", "testname": "switch_test", "prepOptions": "$WORK_TOP/tb/switch/switch_rtl.json,$WORK_TOP/tb/switch/switch_tb.json", "compileOptions": "", "simOptions": "+UVM_CONFIG_DB_TRACE,+atp_driver_rate=30",
    "+NUM_TRANS" : 1000, "+UVM_MAX_QUIT_COUNT" : 5, "+UVM_TIMEOUT" : "",
    "+UVM_VERBOSITY" : "", "iter": "",
    "label":["default", "smoke"] 
    } }

    Here "+UVM_VERBOSITY" & "iter" are not specified. So the script will use the default "+UVM_VERBOSITY" : "UVM_NONE" & "iter" : 1.

  3. The options marked in RED are the TESTCASE names

  4. One underlined GREEN are the ENVIRONMENT(blkTB = Block Test Bench) the belong to.

  5. In DARK BLUE are the LABELS associated to that testcase. It's an array and can have any user choice label.


C) Filtering TestCases using runsim options :

  • runsim filters testcases depending on the filtering options -e / -t / -l.
  • The options provided are ANDed together to run those filtered configs. Refer section A) Custom Help - Table & screenshots on how the configs are filtered.
  • runsim has a Smart Filtering mechanism by using "RegEx" - regular expression.
    For example:

    1. If the user wants to run all testcases with "cfg1" mentioned in testnames, provide the following command:
      runsim -t ".*cfg1.*"                   As seen in the figure below ".*cfg1.*", here the .* will match any string before and after cfg1 (" . " represents any character and " * "represents zero or more of character preceding it)

    2. If the user wants to run two different testcases or two labels simulatneously, provide using the "|" (OR) option of RegEx in the following command:

      runsim -l "checkin|smoke"                 As seen in the figure below, the "|" will match checkin and smoke labels

      ***********DISCLAIMER: ALWAYS USE " " QUOTES WHEN PROVIDING REGULAR EXPRESSIONS***************

  • The filtering mechanism defines an execution flow. All environments should have one "default" label associated to one testcase.
    • So this makes easier for the user in the sense that by just giving an environment option -e, runsim will run that particular "default" testcase.
    • In the following image you can check how each environment has one default testcase.                  And by just saying runsim -e switch, the default testcase will be executed, as shown below


    • If an environment doesn't have a testcase labeled with "default" and the user tries to run just using -e, then runsim will error out with following error: 
      When only -e option is provided, runsim will try to run the testcase labelled "default"
    • If the user wants to run all the testcases under one environment, pass -l all alongwith -e. Refer section D) Running Regressions for example.
        

D) Running Regressions :

  1. The tool enables to run tests on the Sun Grid Engines by providing the -g option.
  2. User can provide option -i <integer> (Iterations), the number passed to -i will generate those many random seeds. 
  3. User can associate lables like "nightly/smoke/weekly" to respective tests.
  4. The tool will dump Regression files in Users scratch3 area. path: /scratch3/USER/sym_regr_mm_dd_yyyy_hh_mm/ . The timestamp format helps in making new directory always.
  5. For example:
    • runsim -e switch -l all -i 10 -g   ( - l all runs all configs in switch blockTB, -g enables grid and -i 10 will run 10 seeds per testcase) 
  6. Label -l all  will run all testcases.
  7. User can also run tACHL Coverage with Regressions. Refer section F) tACHL Coverage Support for more information.
    • runsim -e "switch.*" -l all -i 10 -g -t_cov ( -e "switch.*" will filter switch & any environments starting with "switch", in this case switch_hier; -t_cov enables tACHL Coverage along with the regression)

E) Failure Triaging & Re-run :

  1. The tool features SMART TRIAGING and AUTO-RERUNNING failures based on UNIQUE signature with SMALLEST simulation time.
  2. The Failed Tests will be automatically sorted by their Signatures. The common signatures will be clubbed together and the one with smallest Simulation time will be displayed in 2nd Table.
  3. User can also enable option to Re-run these failed tests. The -r ( Re-run) option will increase the Verbosity to UVM_HIGH and enable Waveforms dump for the user to start debug.
  4. The path to Log and Wave Files are also mentioned in a 3rd Table.
    • runsim -t <testcase> -i 10 -g -r             ( -r option enables ReRun with High Verbosity and Dumping of Waves)


F) tACHL Coverage Support :

  1. The tool enables user to run TACHL Coverage on any environment/testcase just by giving the option -t_cov <string>.  
  2. There are two modes -
    1. Run only tACHL Coverage:     runsim -e <env> -l all -t_cov only
      The "only" string input is required to run just the tACHL Coverage over tests filtered by given options.


    2. Run tACHL Coverage + Simulation:        runsim -e <env> -l all -t_cov
      Don't pass any argument to -t_cov. Refer section D.7) for more information and screenshot.
  3. TACHL Coverage uses pkgMap.json to get all the information required to run coverage. The environment names should match with the names inside pkgMap.json
    If any of the information is not mentioned in pkgMap.json, then runsim will error out displaying the missing information and declare that environment non-coverable.
  4. After successfully running coverage. Runsim will generate 2 folders and 1 file.
    1. The two folders are preped_<env name> & report_<env name>.
      preped_<env name> - contains all the files related to coverage run
      report_<env name> - contains all the report files of the coverage.
    2. runsim also generates a summary_tachl.txt file which consists of the TACHL Coverage Report.
