CONC-9669

HangY

Two reads waking up and beating this new write every arbitration cycle. These two reads cant make progress because there are no DCE credits for it. The DCE is waiting for this Write inside the IOAIU so it won’t release that credit. The solution would be to guarantee that a txn has a credit before its woken up.


Enabling QoS in this case would have fixed the issue.

CONC-9696

Hang or data corruptionY

Write in arb sees enough OTT entries available based off q_oc_count. But, when it goes to allocate into the OTT and actually pick an entry using t_oc_free, t_oc_free is way backed up and there aren't enough free entries in the list to give it. Issue is with the synchronization of t_oc_free and q_oc_count.
