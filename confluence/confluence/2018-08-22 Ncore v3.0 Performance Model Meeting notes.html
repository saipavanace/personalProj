Discussion items

TopicNotes
Communicating design to PA
  • PA with Ncore 2.x used CSV files for unit params, FlexNoC models and topology. PA with Ncore3.x may use CSV, but will need to communicate topology to PA in some other way. When FlexNoC transport mapping is added to Maestro, we will need to have an interface to PA that allows information from both the Maestro topology and the FlexNoC topology.
Customer models
  • Customer models will be supported with a gasket (possibly TLM) to join them to the Maestro models & topology.
Design exploration
  • PA with Ncore 2.x provides design exploration within PA, allowing the user to modify unit parameters and re-simulate. The user can then output a new APF file from PA. Is a similar capability needed from PA / Ncore 3.x, or will it be adequate to say that a fast loop to modify & re-simulate will only be provided in the Maestro native performance simulator?
Multiple simulation platform support
  • We will need to support two different performance simulation platforms (PA vs Maestro native). Which one should we do first? It might be faster / easier to do the Maestro native simulation first because the PA simulation will require coordination and some work from Synopsys.
  • Some APIs will need to be duplicated - one flavor for PA and another for Maestro's simulator, e.g. statistics and logging. We need to support the PA interfaces, but we have an opportunity to provide more information in the Maestro native interfaces.
Postprocess vs interactive
  • Will we do performance simulation batch only, or will we support interaction between the performance simulator and the Maestro cockpit (e.g. activity heat map)?
  • There are limits to what we can do interactively (e.g. due to simulation performance).
  • Interactivity makes for sexy demos.
  • We will be dumping log files. Those could be useful in either interactive or batch.
  • We could support something like 'step through' mode, even in batch mode (log file results are ordered).

  • Could support tagging a message with a secret id that can then be tracked as it makes its way through the models.

  • Need to be careful because of file load times - some files are huge.
TCL API
  • Need a straw-man API format ASAP (for topology creation and ...?).
  • Jeff has some extensions to Moez's work.
    • Moez's work on topology creation was a prototype and will be replaced with TCL access to the ADM APIs.
  • Currently Maestro TCL does not support handles or TCL namespaces. Jeff will make a request for changes he'd like from the Maestro TCL.
  • Split discussion into two parts:
    • TCL command interfaces to the data model.
    • TCL features, e.g. handles to data objects which could allow manipulation of the data using standard TCL APIs.
  • Data model needs to be able to output a Boost graph.
Ancillary features
  • Everyone should look at Jeff's Word document (NCore3 fabric performance model) section 2.4 "Excluded or ancillary features".
  • Mostly contains options for the HW team to decide.
  • Deadlock / livelock analysis will be done statically in Maestro.
  • John - there is a way that deadlock could occur even though the static loop check passes. Something having to do with the way that virtual channels could be set up. In that case, it would be valuable to have deadlock / livelock information from the perf simulator.
  • Can gather switching activity and pass that back to Maestro.
