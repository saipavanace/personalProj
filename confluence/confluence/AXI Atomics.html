Overview

AMBA 5 introduces Atomic transactions, which perform more than just a single access and have an operation that is associated with the transaction. Atomic transactions enable sending the operation to the data, permitting the operation to be performed closer to where the data is located. Atomic transactions are suited to situations where the data is located a significant distance from the agent that must perform the operation.

axsize

The maximum number of bytes to transfer in each data transfer, or beat

The size of any transfer must not exceed the data bus width of either agent in the transaction

if axlen=0, narrow transfers are possible. arsize=0,1,2...log2(bus-wdth)

if axlen>0, wide transfers, entire bus width should be utilized for the transfer

arsize=log2(bus-width)

Narrow bursts are limited to only a single beat

For ATOMIC ops, ATMLD, ATMST,

ATMSWP, Outbound data value size: 1, 2, 4, 8 byte

So, arsize == 0, 1, 2, 3

For ATMCMP - Outbound data size: 2, 4, 8, 16, 32 byte

So, arsize == 1, 2, 3, 4, 5

axburst

burst_type defines how the address for each transfer in a burst changes with respect to previous transfer

INCR, WRAP

FIXED burst_type is not supported

For ATOMIC ops, ATMLD, ATMST,

ATMSWP, burst_type must be INCR

For ATMCMP transactions where the address of the transaction is aligned to the total size of the outgoing data (indicating the Compare value is sent first, followed by the Swap value) the burst type must be INCR.
For ATMCMP transactions where the address of the transaction is not aligned to the total size of the outgoing data (indicating the Swap value is sent first, followed by the Compare value) the burst type must be WRAP.

atm compare

Addtional constraints for Atomic compare

For AtomicCompare:
• The write data is 2, 4, 8, 16, or 32 bytes and read data is 1, 2, 4, 8, or 16 bytes.
• AWADDR must be aligned to a single write data value, half the total write data size.
• If AWADDR points to the lower half of the transaction:
— The compare value is sent first. The compare value is in the lower bytes of a single-beat transaction,
or in the first beats of a multi-beat transaction.
— AWBURST must be INCR.
• If AWADDR points to the upper half of the transaction:
— The swap value is sent first. The swap value is in the lower bytes of a single-beat transaction, or in the
first beats of a multi-beat transaction.
— AWBURST must be WRAP.
For AtomicCompare transactions, there are relaxations to the usual rules for transactions of type WRAP:
• A burst of length 1 is permitted, AWLEN=0
• AWADDR is not required to be aligned to the transfer size

axcache

axcache values define the memory attribute signalling

0 - Bufferable

1 - Modifiable

2 - Read Allocate

3 - Write Allocate

all values from Table A4-5 Memory type encoding

 

For EXCLUSIVE ops-

  1. The value of the AxCACHE signals must guarantee that the slave that is monitoring the exclusive access sees the transaction. For example, an exclusive access must not have an AxCACHE value that indicates that the transaction is Cacheable, i.e AxCACHE[3:2] = 'b00

  2. If there is a buffer or cache which might respond to an exclusive access before it reaches the monitor, then the exclusive access must be Non-bufferable  i.e AxCACHE[0]=0

ATM transactions, permitted to use all legal combinations of AWCACHE and AWDOMAIN, as shown in Table C3-3, AxCACHE and AxDOMAIN signal combinations.

Ncore 3.x supports only cacheable coherent and non-coherent atomics to normal memory connected below a DMI. Ncore
does not support atomic transactions to non-cacheable device memory or cacheable normal memory below a DII.

axid

transaction ID

axid_collision: all transactions issued with same axid

Check that if axid_collision is set, all transactions are issued with same axid

Atomic transactions must not use AXI ID values that are used by non-atomic transactions which are outstanding at the same time. This applies to transactions on either the AR or AW channel. This ensures that there are no ordering constraints between atomic transactions and non-atomic transactions.
Atomic transactions and non-atomic transactions can use the same AXI ID value, provided one transaction has fully completed before the other is issued.
Multiple atomic transactions that are outstanding at the same time must not use the same AXI ID value.

AMBA® AXI and ACE Protocol Specification ARM IHI 0022H.c

A4.8 Legacy considerations

In AXI4, all Device transactions using the same ID to the same Subordinate must be ordered with respect to each other.
