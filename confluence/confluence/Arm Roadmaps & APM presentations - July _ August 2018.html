NOTES

ML “Marvin” safety wrapper /  FlexNoC integration (Andrew)

  • wrapper + BIST using FlexNoC
  • Goal: Quick marketing win
  • ASIL B
  • ACE-lite I/F
  • Proposed technical solution: 2 ML cores run in lockstep with same inputs, each connected to an Arteris FlexNoC NIU. NIUs connected to checkers within interconnect, which are then connected to Fault Controller
  • Can be stepping stone to fault isolation capability proof of concept

Fault management architecture (Andrew)

  • Fail operational: Best technical approach is fault isolation and partial reset of offending /affected connected IPs and parts of NoC
    • TMR too costly; multiple paths too costly
  • Joint Arm/Arteris interest in clearer (or maybe even tighter) integration to deal with the system-level issues of partial system reset

AMBA Update (Bruce)

  • CHI Issue D
    • Memory Partitioning and Monitoring (MPAM) – SW-visible QoS-style tagging for VM allocation
      • May affect Ncore CMC and CodaCache?
    • 2-part Persistent Cache Maintenance Operations (PCMO) – differentiate between visible/cleaned and persistent
    • DVM changes – early completion signal, less constraining
    • Performance hints – probably implemented in Issue E but could be Issue D
      • System-level cache policy hints
      • Impact on Ncore CMC?
  • CHI Issue E – alpha availability late 2018
    • CPU arch-inspired updates, probably no effect on interconnect
      • DVM – linkage to OS and CPU
      • Transactional Memory – for database-type applications
    • Assign multiple interface ports per IP – Info avoids putting 2 addresses into snoop filter
    • Replicated channels – wider data channels
      • Some things address-only, or to deal with multiple responses
    • Direct write to system level cache
    • Direct write transfer
  • AXI and CHI coalescence
    • Will be AXI Issue G in 2021
    • Deal with painful ordering and interleaving
    • Marketed as “next gen AXI”

Cortex-M Update (James, Andrew)

  • Current Cortex-M33 uses AHB5 with TrustZone (PSA security)
  • New Cortex-M33P adds new features for automotive
    • Every flop has one or more parity bits – for security and FuSa
    • ECC for bus I/F is user configurable, but Arm will push customers to align with Cortex-R implementations
    • Can be used DCLS
    • Will have safety package (Safety Manual, etc.)
  • “Yamin” adds MVE vector extensions and AXI5-M interface.
    • What subset of AXI5?
      • Andrew to share I/F spec when available
      • Arteris IP could use Carbon model of “Yamin” or Cadence VIP
    • AXI5 TrustZone is different than regular TrustZone!
      • MPU mechanisms
    • Opportunities for joint marketing: Higher performance with very low power

Cortex-R Update (James, Tom, Andrew)

  • Current Cortex-R8 (Armv7) is 4-core cluster capable. Used in modems and storage.
  • Armv8-R64 products: “Fuji” and “Kinabalu”
    • 64-bit processing only
    • AMBA5 CHI master
    • Up to 8-core cache coherent cluster; CHI master cluster port 128-bit wide; 64-bit proc only (no 32); AXI5 LLRAM port (128-bit) for non-coherent low latency accesses; ACP
  • Cortex-R5 still popular in automotive. Is cheaper.

Cortex-A Update (Andrew)

  • New cores are “Enyo”/Cortex-A76 (launched), “Helios” (at LAC), and “Deimos” (at LAC)
  • “Split Lock” capability allows 2x DCLS or 4x cluster
    • “Enyo-SL”, etc,
  • Early launched version (R0) have not I/f protection or SL capability. R1 versions will add these

Trillium ML and OD (Andrew)

  • “Marvin” neural processing unit. Called “ML”.
  • “Minsky” object detection processor. Called “OD”.
  • S/W is biggest issue to using ML “Marvin” in multi-cluster, hierarchal architecture
  • Arm marketing goal: show how to implement ML IP in automotive (for safety)
    • Opportunity to “wrap” Marvin in Arteris IP FuSa mechanisms (similar to DCLS) – joint blog article on Arm site, ArmTech Symposia presentations, white papers
    • Opportunity to implement in proof-of-concept / reference design
