CAIU0 standalone back to back test

1) Observation:

For the read standalone test, the bottleneck is NOT from resource/credit any more in this case. Because we have two DMIs,  the BW expected for each DCE is increased. 

Since we are using 100 cycle SRAM model (77 ns), 46 OTT size cannot cover the full BW → Increased ATT size and compared the results. 

  • One DMI case: cannot achieve the performance because ATT is full. DCE cannot generate back to back traffic. 
  • Two DMI case: Observing back to back transaction between cmd/mrd request in DCE when using 64 OTT size. (Still need to find out 7%!!! where !!)
  • For the write standalone test, RB credit is relatively high but not stressing the system yet.

CAIU0/CAIU1 back to back test

Used readOnce to simplify measurement - to begin with basic test. 

1) Observation:

46 DCE case, Two DMI, Read

  • Read back to back shows < 45% of the performance with 46 DCE
  • Showing the similar behavior with CAIU0 standalone case - system is congested by DCE ATT.

64 DCE case, Two DMI, Read

  • Read back to back shows < 78% of the performance with 46 DCE
  • 80% is not bad considering all miss and 64 ATT size. 
  • Still observing MRD is not generated in a back to back way because of the ATT credit, but the percentage is reduced. 

46 DCE case, Two DMI, Write

  • Write back to back shows 70% performance
  • Bottleneck is from RBID credit

64 DCE case, Two DMI, Write

  • Write shows same performance, because the bottleneck was not ATT outstanding capability.
