  • End-to-end credits: will be required for Ncore 3.0. Two independent end-to-end mechanisms; one for Ncore units (which we need for Ncore 3.0) and one between Symphony ATUs (only needed for later releases).
  • Latency estimation pre-simulation: needed for 3.0. Ncore team willing to annotate that as a CPR function which will populate ADM; ADM subsequently will know how to estimate latencies.
  • Power domains: two scenarios needed:
    • Power-down of whole Ncore system (for mobile). Requires a P-channel towards the outside but no special handling logic.
    • Power-down of individual processors. Requires the ability to define domains. Depending on where the domain boundary is set (north or south of AIU) different solutions needed. Current proposal is to put boundary outside the AIU, which is easier. More details to follow.
  • PPA estimation: power very hard, but area is very important. Will need to rediscuss characterization progress.
  • Adapters: clock, width and pipe adapters sufficient for Ncore 3.0. Others (e.g. power, VC and firewalls) later.
  • Interrupts: only need to handle a few outgoing wires that need to be brought from units to toplevel interface. Maestro will not need to optimize them at the physical level. Only requirement will be to add a timing constraint (e.g. set_input_delay) in a generated backend script.
  • Memory macros: needed for more efficient implementation of Ncore units, e.g. DCE. Maestro and ADM to provide minimal support, the flow will rely mostly on Verilog wrappers on disk. Need for simple backannotation of area metrics into ADM though; can initially ask for an area number, and then maybe DEF import.
  • Grouping: this feature is different from IPG/subsystem composition. Main usage model is to break up a big netlist in smaller chunks, more usable for backend flows. Decisions: only one level of grouping needed; grouping happens by breaking the design along links.
  • Undo: needed for Ncore 3.0.
