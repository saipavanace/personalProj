Discussion items

TimeItemWhoNotes
1hrCCTI and CSTI requirementsmonica tang (Deactivated)
  • POR is Q channel support only. If we need P channel support, will need to go through change request process. CCI, CCN all offer P-channel support.
  • ARM is coming up with a power controller. We need to ensure our P/Q channel interfaces are compliant to their IP
  • Resilience features in the transport (ECC/parity/packet checks/timeouts) are used by customers - other than unit duplication. Need to understand the intended coverage and what is required in Symphony transport
  • Memory interleaving and reorder buffering is not required in the transport interconnect. Ncore units will take care of the interleaving.
  • For performance improvements, will need to transport command and data separately (rather than multiplexing into the same links). Need to investigate whether it is more efficient to have 2 separate CCTI networks or 1 single CCTI but with 2 connections to Ncore units
  • Possible to have read/write transactions in Ncore native interface for Ncore3
  • What does QoS in Ncore mean? (what are the benefits? what are the implications to the design?)
  • Debug - need to have "hidden" debug capabilities
  • Topology editor - needs to be "regular topology friendly" - i.e. should be easy for user to create regular topologies
  • POR - automatic pipeline insertion is not required for first release
  • Ncore3 is now going to be one product - should consider merging functions, rather than having distinct functions for transport vs Ncore units
  • CSTI: Ncore units currently take in OCP-Lite protocol. Consider moving it to APB such that Symphony does not need to support OCP Lite in the first release
