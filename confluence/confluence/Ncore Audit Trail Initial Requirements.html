Audit Trail

  • Requirements for Ncore
  • alexis boutillier (Deactivated), JeanPhilippe Loison, benoit de lescure (Unlicensed)

Purpose

  • This presentation list the set of requirements for a improved tracking of the usage of our IP in customer chip
    • Help sales with contract enforcement
    • Help sales with contract renewal
  • A complete set of requirement is presented
    • Implementation of features over time to be discussed

 

Definitions

 

  • PROJECT: a project is how Arteris define one “use” of its IP. Typically it is a single chip design for the Customer.
    • Projects are linked to a particular contract between Arteris and the Customer
    • Projects have a start date (when customer starts using the tool to begin the design process) and a finish date (when chip is taped out)
  • CAPTURE: use of the Arteris tool to specify the IP configuration. The result of this action is in a .apf file
  • GENERATION: creation of the RTL implementation of the IP by the Arteris tools.

Tracking

  1. It must be possible to link unambiguously a particular .apf file with a particular PROJECT
  2. It must be possible to link unambiguously the files containing the RTL code GENERATED by the tool to a particular PROJECT
  3. It is desirable to be able to link unambiguously a design in net-list form (after synthesis, place & route) to a particular PROJECT

Control “a posteriori”

  1. It shall be possible to obtain easily from customer a summary of all the GENERATIONS done by the tool over a given period of time
    1. A log file containing information regarding the IP configuration, and the related PROJECT shall be maintained automatically
    2. It shall not be possible to tamper the log file
    3. The log file shall be readable by the customer and Arteris
    4. Activity from multiple users of the tools but related to the same PROJECT must be in the same log file
    5. It shall not be possible to produce different log files for the same installation of the tool / same license server
  2. It shall be possible to exploit automatically the content of the log file
    1. Extract information easily
    2. Definition and detection of “alarms”

Control “a priori”

  1. It shall be possible to check if a GENERATION performed by the customer is legal
    1. For every single GENERATION
    2. For a GENERATION tagged as “final” (the last GENERATION that will produce the RTL that will go onto the chip)
  2. It shall be possible to allow / deny a GENERATION request
    1. For every single GENERATION
    2. For a GENERATION tagged as “final” (the last GENERATION that will produce the RTL that will go onto the chip)
  3. It shall be possible to deliver to a customer a tool with no capacity to GENERATE a usable RTL
    1. The GENERATION will be done by the customer, but RTL will be impossible to use unless Arteris deliver a Key for that particular GENERATION
    2. The GENERATION will be done entirely by Arteris for the customer

Automation

  1. Arteris shall implement a database to track PROJECT status
    1. Sales and AE need to be able to create and modify PROJECTS and related authorizations
    2. Project activity gathered from customers must be logged in the database
    3. PROJECT activity need to generate notifications to stakeholders
    4. Sales and AE need to be able to query the status of projects
    5. Finance need to be able to query the status of PROJECTS
  2. Communication of information between the Customer and Arteris need to be automated
    1. Client (tool given to customer) Server (at Arteris) communication model
    2. Email as a backup solution
