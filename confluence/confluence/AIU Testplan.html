2.2.1 Monitors

ACE/AXI4/ACE LITE Monitor (dv/common/lib_tb/axi_monitor.svh)

This monitor looks at the following interfaces between the ACE BFM (or one of the ACE VIPs) and the AIU DUT:

  • Ace Read Address Channel
  • Ace Write Address Channel
  • Ace Read Data Channel
  • Ace Write Data Channel
  • Ace Write Response Channel
  • Ace Snoop Address Channel
  • Ace Snoop Response Channel
  • Ace Snoop Data Channel

The monitor can be configured to be monitoring an AXI4 or a ACE Lite interface as well, in which case it will only look at a subset of the above channels (channels which exist are described by ARM AMBA Ace spec).

 

On the data channels, the monitor will collect all beats of data before sending a complete packet with all the data.

SFI Monitor (dv/common/lib_tb/sfi_monitor.svh)

The SFI monitor looks at the SFI master and slave request and response interfaces between the AIU and the system BFM.

AXI4 Monitor (dv/common/lib_tb/axi_monitor.svh)

This monitor looks at the AXI channels (subset of the above 8 channels) between the AIU and the AXI4 VIP/AXI BFM.

2.2.2 BFMs

ACE BFM

The in house ACE BFM will be used when the 3rd party VIP is not being used.

System BFM 

The system BFM will act as the rest of the system to the AIU DUT. It will process requests sent by the AIU and will send snoops to the AIU for addresses previously requested, currently in process or random addresses. 

2.3 Checkers

AIU Scoreboard

 

The AIU scoreboard monitors all the external signals coming in and out of the AIU DUT. It is fed all of this information by the ACE/AXI4/ACE LITE monitor (which monitors signals between the ACE BFM/ACE VIPs and the AIU), the SFI monitor (which monitors signals between the BFM and the AIU) and the AXI4 monitor (which monitors signals between the AXI4 VIP/AXI BFM).

 

The AIU scoreboard performs interface level checks between the ACE/AXI4/Ace Lite side and the SFI/AXI4 side. Here are the checks it performs:

  • Basic ACE transaction errors like illegal AxSnoop type.
  • Checking the transaction flow matches the flow as described in Figure 1 for a ACE Read (Make/Clean), ACE Write (Evict) and Concerto Snoops
  • Check to make sure that the ACE VIP/BFM sends the correct axlen field (this cannot be more than the data bus width).
  • Check to make sure incoming and outgoing transactions have exactly one matching possible transaction based on different conditions. For example, an ACE read data with a particular rid should match only one ACE read address channel transaction with the same arid for which data has not been received and which has not been blocked because it wasn’t behind another ACE read channel transaction with the same arid.
  • Check all ACE fields in the follow up ACE transactions (ACE Read Data channel fields are compared to ACE Read Address channel fields to make sure they match)
  • Check to make sure that ACE fields match SFI fields received on ACE transactions that follow SFI transactions. For example, an ACE snoop address channel transaction should have the same protection field as the incoming SFI snoop request message).

    ·       Check some ACE rules for snoop responses:

    o   PassDirty=1 and DataTransfer=0 is illegal

    o   For ACE commands RdUnique, CleanInvl and MakeInvl, IsShared bit should be set to 0.

    ·       Check ACE snoop type matches SFI snoop message type.

    ·       Check ACE read/write response field to see that ExOkay response is received only when request type is an exclusive request.

    ·       Check to make sure that SFI fields on outgoing SFI messages match SFI fields received on incoming SFI messages. For example, an outgoing DTR request that follows an incoming SNP request should have the same SFIPriv AceUser field.

    ·       Check to make sure that SFI fields on outgoing SFI messages match ACE fields received on incoming ACE transactions. For example, an outgoing DTW message should have the same data as an incoming ACE data on the write data channel.

    ·       Making sure that SFI transaction IDs for SFI outgoing request packets match the transaction ID requirements (Trans ID prefix) as laid out by the Concerto AIU architectural spec.

    ·       Making sure that SFI transaction IDs for SFI outgoing requests are unique and do not match any SFI requests for which SFI responses have not been received.

    ·       Check to make sure that outgoing SFI requests (and incoming SFI requests) have the correct SFIPriv message type. The only SFI requests that are outgoing are CMDreq, DTWreq and DTRreq (only for snoops). The only SFI requests that are incoming are SNPreq, DTRreq and STRreq.

    ·       Check to make sure that incoming SFI snoops have unique pair of SFIPriv AIUID and SFIPriv AIUTransID. Also check to make sure that this snooping AIU is not the same as the requesting AIU. This check only applies to incoming snoops of type SnpClnDtr, SnpVldDtr and SnpInvDtr (others could be caused by recall, for which AIU ID/TransID is don’t-care).

    ·       Check to make sure that the exact number of SFI DTR request messages are received (based on CMD request type and/or STR request ST field).

    ·       Check to make sure that, if multiple DTR request messages are received, the data in them is exactly the same.

    ·       Check DMI and DCE unit IDs on Concerto message requests going to those units based on the address.

    ·       Check SFI snoop response correctly reflects the ACE snoop response.

    ·       Check SFI NonPriv fields based on tables 21-28 in the Concerto AIU architectural spec.

    ·       Check to make sure that the STRReq and STRRsp messages are legal based on the CmdReq message.

     

    The following are a set of checks that still need to be added to the AIU scoreboard:

    ·       Checks for barrier transactions.

    ·       Checks for DVM messages.

    ·       Checks for error conditions.

    ·       Calculation of DCE/DMI unit ID based on address.

    ·       AXI checks (for non-coherent transactions that do not go on the SFI interface).

    ·       Handling partial writes/multiple cache lines (including different data bus widths between ACE and SFI as well as different cache line sizes between ACE and SFI).

    ·       Setting up DTW requests for read transactions that end up needing to write data to memory.

    ·       Snoop requests that end up sending DTW data.

    ·       Check to make sure that an exclusive load should always get an EXOKAY = 1 response if exclusive accesses are supported.

     

    System Verilog Assertions

    This section will include the following sets of assertions that will be added to the testbench:

    ·       3rd party VIP assertions for ACE/AXI4

    ·       Assertions available on arm’s website to check ACE/AXI4 behavior: These assertions are standalone that will be alive in simulations in which the ACE in house BFM will be used.

    ·       Interface (external as well as between two internal blocks in the AIU) assertion checks that would make more sense to be written as assertions rather than checks as part of the AIU scoreboard or micro-architectural checker.

     

End of Sim Checkers

·       Check to make sure all transactions have completed from ACE BFM and system BFM perspective.

·       Check to make sure all credits have been returned at the end of simulation and credit counters are back to reset values.

·       Check to make sure all AIU internal queues/structures (OTT Ctrl, OTT Data, OTT Strb, STT, DSB, DMB, DCF and possible others) have no pending transactions.

 

·       Check to make sure all AIU state machines are back to “idle”.
