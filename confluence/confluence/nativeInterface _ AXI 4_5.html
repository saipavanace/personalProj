    axaddr

    addr will be covered in the following distinct bins-

    1. cache-aligned (ex. for 64B cacheline-size, addr[5:0]=0)

    2. cache-unaligned && burst-size aligned

    (ex. for 64B cacheline and 16B bus-size, addr[5:4]!=0 && addr[3:0]==0

    3. cache-unaligned && burst-size unaligned

    (ex. for 64B cacheline and 16B bus-size, addr[5:4]!=0 and addr[3:0]!=0)

    randomized in address_manager based on address regions:

    coherent addr

    addr_mgr.get_coh_addr

    non-coherent dmi addr

    addr_mgr.get_noncoh_addr

    IO coherent/DII addr

    addr_mgr.get_iocoh_addr

    For WRAP burst: The start address must be aligned to the size of each transfer.

    For INCR burst: Completely unaligned start address possible

    For EXCLUSIVE ops- address must be aligned to total number of bytes in the transaction.

    ex. for a 128B exclusive, addr[6:0]==0.

    EXCLUSIVE ops are always non-coherent RDNOSNP, WRNOSNP (ie. issued from addresses that map to GPRAR.NC=1 region)

    axlen

    Burst-Length indicates number of data transfers or beats

    Burst_Length = axlen + 1

    For INCR: 0-255

    For WRAP: 1,3,7,15

    For EXCLUSIVE ops, following should hold true: axlen <= 15

    (2 ** axsize) x Burst_Length <= 128B

    axsize

    The maximum number of bytes to transfer in each data transfer, or beat

    The size of any transfer must not exceed the data bus width of either agent in the transaction

    if axlen=0, narrow transfers are possible. arsize=0,1,2...log2(bus-wdth)

    if axlen>0, wide transfers, entire bus width should be utilized for the transfer

    arsize=log2(bus-width)

    Narrow bursts are limited to only a single beat

    For EXCLUSIVE ops, following should hold true:

    (2 ** axsize) x axlen <= 128B

    axcache

    axcache values define the memory attribute signalling

    0 - Bufferable

    1 - Modifiable

    2 - Read Allocate

    3 - Write Allocate

    all values from Table A4-5 Memory type encoding

    For EXCLUSIVE ops-

    1. The value of the AxCACHE signals must guarantee that the slave that is monitoring the exclusive access sees the transaction. For example, an exclusive access must not have an AxCACHE value that indicates that the transaction is Cacheable, i.e AxCACHE[3:2] = 'b00

    2. If there is a buffer or cache which might respond to an exclusive access before it reaches the monitor, then the exclusive access must be Non-bufferable  i.e AxCACHE[0]=0

    axqos

    AxQOS is used as a priority indicator for the associated write or read transaction. A higher value indicates a higher priority transaction.

    randomize to the following values by default

    0b0000 - 5% (interface does not participate in any qos scheme)

    0b0000 - 0b1111 - 95%

    user can specify a deterministic value.
