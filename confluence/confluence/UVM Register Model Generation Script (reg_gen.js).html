1. Introduction

Problem Statement

As of today, maestro does not have the ability to generate a UVM register model. We typical rely on external tools to generate the register model like ral_gen from synopsys or csrCompiler (Arteris internal tool), however, these tools cannot be directly integrated into maestro since they are software builds (binaries) and not just a piece of code. We would need to access the source code of these tools to integrate them into Maestro. For tools like ralgen, as they are proprietary, we cannot access the source code and for our internal tools like csrCompiler, even though we can access the source code, as they are not written in native javascript, it gets complicated to integrate them into Maestro directly (even though this is worth exploring).

Because of this, we relied on our customers to have a tool like ralgen which they need to run on their environment so that we can generate the UVM register model from the IPXACT that we provide. This raises two issues:

  • First is that, this puts additional burden on the customers to install ‘ralgen’, point to the installation directory, setup the flow and then run ralgen before they can run any simulations

  • More importantly, not all customers might have 'ralgen', like our Cadence only customers which already led to some problems with our customers

We could even try asking our customers, even though cumbersome, to run whatever tools they access to. Like reg_verifier tool from Cadence for Cadence customers and ralgen for synopsys customers. However, using reg_verifier is not a straightforward process which requires multiple steps like downloading AMBA bus definitions, installing reg_verifier, setting up the tool, etc

Solution

Given the above problems, the best solution is to have a simple script written natively in javascript which can be easily integrated into Maestro and so when our customer generate the RTL and DV, they automatically get the UVM register model which is generated by Maestro with the help of this script.

Since its out internal tool which generates a standard System Verilog register model, we will be completely agnostic to the toolchain used by our customers, even for future customers with a completely new toolset

2. Script

parseXML.js

The script takes an IPXACT file as an input to generate the UVM register model. Therefore we need to parse the IPXACT file (which is in XML format) to construct an object in javascript which can be used easily to generate the UVM register model. We could have used any external javascript libraries like ‘xml2js' but this increases our maintenance in the future whenever we have to upgrade the library and we have to deal with any breaking changes within the packages including their potential support abandonment.

To avoid that, we have our own script to parse the XML structure which is done with the parseXML.js script

This script is located here: http://gitlab.arteris.com/hardware/hw-ncr/-/blob/Ncore3.7/develop/dv/scripts/registerGen/parseXML.js

reg_gen.js

This is the script which uses the parseXML.js script to parse the input IPXACT file and constructs the UVM register Model. This is then dumped out in the output path specified with a default name

This script is located here: http://gitlab.arteris.com/hardware/hw-ncr/-/blob/Ncore3.7/develop/dv/scripts/registerGen/reg_gen.js

3. Usage

Integration into another script

The reg_gen.js script can either be called stand-alone on the IPXACT or be integrated into another software (like Maestro).

  • The script exports this function: generateUVMRegisterModel

  • This is the script Usage:

    • generateUVMRegisterModel(xmlFile, outputFilePath, ncoreModelName, ncoreFileName, fscModelName, fscFileName);

  • Where, xmlFile is the IPXACT file (can be either 2009 or 2014 - 2009 preferred)

  • outputFilePath is the path where the register files should be dumped. Example: “scratch/spavan/myProject/”. Default value is “./“, meaning if we dont pass in anything, we will dump the output file in the location where this function was called from

  • ncoreModelName is the name of the register model class, which is by default “ral_sys_ncore“

  • ncoreFileName is the file, in which all the UVM register model classes are written. By default this is “ncore_system_register_map.sv“

  • fscModelName is the name of the register model class for FSC registers, which is by default “ral_sys_resiliency“

  • fscFileName is the file in which all the UVM register model class for FSC registers are written. By default this is “ncore_fsc_system_register_map.sv“

Running the script stand-alone

We can run the script directly by passing the IPXACT file, if we want to generate a UVM Register Model quickly.

Usage: reg_gen.js -ipxact <path\to\ipxact2009|2014> [options]
Options:
-ipxact <path> Path to the IP-XACT file (2009 or 2014 version)
-o <path> Path where output files should be dumped (default: ./)
-ncoreFileName <name> Name for the ncore output file (default: ncore_system_register_map.sv)
-fscFileName <name> Name for the fsc output file (default: ncore_fsc_system_register_map.sv)
-ncoreModelName <name> Name for the ncore model (default: ral_sys_ncore)
-fscModelName <name> Name for the fsc model (default: ral_sys_resiliency)
-h, -help, --help, --h Display this help message
