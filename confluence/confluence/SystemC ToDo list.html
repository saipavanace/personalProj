NCORETLM TODO (AXI/ACE TLM Support)

Remaining items: 

  • Additional phases on TLM interfaces to support backpressure. 

    • Presently, the TLM interfaces have a limited number of phases. Although the xactors can tolerate some back pressure at the pin interface, too much back pressure propagates back to the TLM interface and the model fails. Solution: Implement standard 4-phase model to split responses in response to back pressure. 

  • Specify and define precise definitions of TLM generic payload structure for each transaction type. 

    • Very important to the end user. All additional payload extensions have to be tested and documented. Transaction mapping to standard Generic Payload fields needs to be defined, documented and tested. 

  • Verify TLM protocol compliance.

    • Presently, we have no way to verify TLM protocol compliance. We support the 'Base Protocol' which is the standard-defined protocol. I believe prior models supported only a limited subset of this protocol, which in itself is not compliant. We really should, however, have some confidence that the protocol can be verified.

  • Specify behavior of transactions in the absence of 'our' extensions. 

    • Presently, the code will be heavily reliant upon our own custom GP extensions. We need to define the behavior of the xactor when using a back Generic Payload in the absence of such extensions. NOTE: This may not be entirely possible. In the case of ACE, our extensions are mandatory. 

  • Additional testing using sophisticated (and standard compliant) stimulus to verify behavior. 

    • Testing has been carried out using stimulus that I am generating. Although randomized and self-checking, this may/may not be truly protocol compliant. In addition, it may not be sophisticated enough to find bugs that would manifest in a real system. 

  • 'Smoke' test environment bring-up. 

    • The current code is somewhat rudimentary and somewhat limited (although individual's definitions here vary). More time and development will be required to make it production quality. 

  • Additional 'ncore' configurations to test parameterizations. 

    • I have available to me only one test configuration. I will need a representative set of user configurations that I can build and test. Although basic parameterizations will probably work (differing port lengths), parameterizations differing substantially from the current cfg. could very possibly break things.  

  • Support for AXI-Lite. 

    • No support for AXI-Lite at present. This is a variation of the AXI protocol therefore the standard xactor may be used with some additional parameterization (unknown). 

  • Support for out-of-band pins originating from ncore instance. 

    • Ncoretlm.js script does not consider non-AXI/ACE pins originating from the ncore instance. These will need to be brought out of the module and tied off in the testbench. 

  • Support for multiple clocks/resets. 

    • The JSON I have has only one clock and reset. I believe the mechanism by which this can be inferred will need to be defined.

  • Build support rationalization as integrated in JS environment. 

    • Presently, I generate a CMakeLists.txt from the JS that can be used to generate the smoke-test TB. This may not be sufficient; requirements unclear. 

  • Generation of non-OSCI tool support scripts. 

    • Presently, I support only the standard OSCI/Accellera SystemC distribution. Additional support for QuestaSim/VCS/Platform Architect may be required. The work required to port ancilliary scripts is presently unknown; requirements unclear.

  • Support user-defined NCore instances.

    • Additional parameterizations in code, the mechanism by which this is done is TBD (probably #define).
