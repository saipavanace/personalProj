2.3 QOS feature and Latency checker

QOS feature will not change the functional behavior / protocol of DMI and no new functional checks proposed. The intent of the QOS feature is to ensure that the DMI functionality is not broken by exercising various combinations of high priority and low priority traffic proposed in the DMI QOS Feature Stimulus combinations section.

Good to report the average latency for High priority and Low priority traffic as a function of WTT / RTT occupancy to check the effectiveness of the feature implementation.

2.4 DMI deadlock due to lack of RB (Request Buffer) credits from DCE.

The main goal here is to make sure such scenarios don't exist in our network. This requirement originated from a testbench micro-architecture violation. The violation was strictly related with transactions made of 2 sets of data, the architecture described them as primary and secondary data. For example,  CmdWrUnqPtl is one of those types.  In this particular case, DMI had received the primary data before the DTWRsp of the secondary data was even sent out.  This caused a deadlock, as DMI did not expect to fill up 2 data buffers using 1 credit. There is a specific sequence of events that need to occur for such type of transactions. To understand the actual flow of CmdWrUnqPtl, CmdWrStshPtl or CmdWrAtm transaction in the Arch Spec or may refer to Table (Transactions flow for 2 sets of data processed).

2.5 Possible deadlock in DMI when response channels are shared

To understand such deadlock scenario, it would be best to refer to the "DMI 3.4 Read Data Buffer implementation document" ( DMI Possible deadlock with shared AXI rd/wr response channels ).  DV goal here is to make sure that such behavior will not be encountered after verification is done, whether the following parameter 'useMemRspIntrLv' is configured or not. Such parameter will be implemented by the RTL in the TCL files.

  • useMemRspIntrLv = 0:  indicates that the read buffer module will not be instantiated, as long as the AXI target supports read data interleaving

  • useMemRspIntrLv = 1:  indicates that the read buffer module will be instantiated, as long as the AXI target supports read data interleaving

Testcases that will be used to verified this deadlock scenario will have to set the following parameter:: "+k_ace_slave_read_data_interleave_dis".  Here is the definition of such parameter setting::

  • "0"  :  enables read data interleaving

  • "1"  :  disables read data interleaving

 Notes:: DV and RTL teams agreed on not adding extra configurations to fully verify such implementation. Instead, we will modify the following existing configurations "config1, config6 and config8" to support the new requirements.

3.1.5 Scratchpad memory

DMI can be configured with or without cache at compile time.  The cache inside the DMI can be configured to have N ways given to Scratchpad memory and rest to the cache memory (where N is greater than 0 and less than the number of ways in a set).

Scratchpad memory behaves same as a normal memory that has N addresses (where N = number of sets * number of scratchpad ways) on which write and read operations can happen. These addresses will be different than those of cache addresses and by using the address, it can be determined if it is scratchpad or cache address.

All the command types that can come to cache memory (described in the above sections 4.1.1 to 4.1.5) can come to the scratchpad memory as well.

3.5.2 Latency counter :

Ncore provides latency counters DMI for either reads or writes. Latency is reported back as a binned
histogram. The latency reported at DMI is more of a closer representation of latency seen at the Native AXI interface.

When transaction is ready to be issued on the Native interface, it is allocated to the latency counter table.

The ID represents the transaction table (OTT/WTT/RTT) ID and counter is a 9-bit counter. The counter is started as soon as the entry is allocated. If all entries within the latency counter table are taken, then any new qualified transaction just does not participate in the latency histogram.

The counter increments every 2, 4, 8 or 16 clock cycle based on CSR configuration. If the counter hits its max value, then it stays saturated until deallocation and the saturated value is used for latency histogram binning.

The latency counter table is deallocated at approximately the same time corresponding response is received from the native interface. The deallocated entry counter value is then subtracted by the configured 8-bit offset, if the offset corrected value is negative then it is saturated downwards to zero. The quantizer takes the offset corrected value and increment appropriate histogram bin.


3.5.3 Latency counter verification strategy

The latency table verification strategy consists of spying on the "alloc" and "dealloc" signals with a latency spy interface.
These signals will be the inputs of the latency counter scoreboard.


Latency scoreboard predicts and generates the different histogram bins
After that, Those bins will be sent to the Pmon scoreboard which will increment the appropriate bin and map each counting value to the correct register.
The 8 registers are checked at the Pmon scoreboard level.


In order to improve the quality of our verification and to be able to precisely identify and determine the location of any bugs, the latency table verification would be on two levels:

1 - The verification of the generation of the bins to detect a possible bug at the level of the latency counter table or offset logic  : we will spy the bins signals of design at the output of the quantizer and will be compared with the bins generated by latency counter scoreboard at the level of this scoreboard.
2- Verification of the bins counting and mapping of values to the different registers at Pmon scoreboard level.



3.5.4 Bandwidth counters

Ncore provides Bandwidth counters DMIs. These counters count in 64Byte data accuracy and at divided by 16 accuracies of the clock frequency.

BW counting can be filtered based on FunitID or user bits, this can be configured in CSRs xBCNTFR and xBCNTMR.

In DMIs the count following : 
• Read data bandwidth, this refers to data being read from DMI : DtrReq event
• Write data bandwidth, this refers to data being written into DMI : DtwReq event

The counter reports : 
• xCNTSR: number of divide by 16 clock cycles
• xCNTVR: number of 64Bytes of data.
The above information can be used to calculate the effective BW. The counter must be disabled before reading them to get correct BW.
Note that this will be approximate BW as all transactions that were counted may necessarily not be 64 bytes transactions and the time accuracy is divided by 16.
