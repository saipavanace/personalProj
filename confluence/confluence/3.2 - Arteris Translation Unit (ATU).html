Overview

The ATU connects a Master/Slave Native Protocol Agent at the periphery to the Symphony interconnect via the ATP interface. The Initiator ATU converts native protocol transactions into packets and injects them into the Interconnect. The target ATU converts these packets back into native protocol transactions and sends them to the Native Protocol Agent. 

The Native Protocols can be:

  • AMBA 5/4 AXI
  • AXI Streaming
  • OCP
  • APB
  • AHB
  • Common Layer Protocol

The ATU is constructed from three layers: the native layer, the common layer, and the transport layer. Below shows these three layers and where each pipe stage would be located.

Initiator vs Target

There are two specific types of ATUs. An initiator ATU (ATUi) and a target (ATUt). The ATUi sits across from a master native interface and takes in requests. The ATUt sits across from a slave native interface where it forwards requests and receives responses to send back to the ATUi . 

These two blocks perform specific roles in the life of a transaction:

ATUi:

  • Partial Address Map
    • It contains specific information about targets which informs the initiator how to handle an instruction.
  • Target/Route Lookup
    • Through the PAM the ATUI will find a destination for a transaction and route it there.
  • Request Striping and Splitting
    • Based on either PAM requirements, or individual ATUi requirements the ATUi may split a transaction.
    • In addition, the ATUi has the ability to stripe a transaction over multiple targets.
  • Queue Mapping
    • The ATUi is able to categorize transactions into separate queues. These queues then can be rate-limited and QoS remapped.
  • Request Buffering
    • The ATUi has the ability to buffer transactions inside the queues.
  • Queue Arbitration
    • Different arbitration schemes can be used to select the order in which queues send out requests.
  • Rate / Request Limiter
    • Each queue can have a rate limit. When a queue reaches its rate limit it can either be downgraded in QoS or be ineligible for selection.
  • Credit Manager
    • TODO: Not in P0
  • Response Reorder Buffer
    • The ATUi reorders responses so all ordering conventions of the native interface are followed.
  • ID Compression
    • The ID can be compressed to a smaller width to save wires inside the network. 

ATUt:

  • Data,Burst, and Address Translation
    • The ATUt has the ability to translate requests from one native interface to another. The differences in these native interfaces can be data width, burst types, etc.
  • Exclusive Monitoring
    • An exclusive monitor can be added inside the ATUt if the native slave cannot handle exclusive transactions.
  • Request/Response Queues and Arbiter
    • Responses from the slave interface can be arbitrated to give priority to one native master interface over another.
  • Credit Manager
    • TODO: Not in P0

Rate Limiter Description and  Parameters

There are buckets (parametrized depth) –

1)      Global bucket -  tracks the request rate for all input queue

2)      Queue Primary Bucket - tracks the request rate per queue

3)      Queue Secondary Bucket - tracks the request rate per queue for a secondary demoted priority 

Each bucket will have a maximum byte count as defined by their width.

Each bucket count represents a number of bytes that a queue has to send.

Each bucket will be given a certain amount of byte credits after its refresh rate.

  • Queue Buckets: refreshAmtQueue[queue]
  • Global Primary/Secondary Bucket: refreshAmtGlobal

For every request selected by the arbiter, the corresponding bucket is decremented. The updated bucket depth does not have to be ready the cycle after arbiter selection – can pipeline the update but during that pipeline cycle the bucket is considered empty (or if the bucket depth is below Delta + Transaction Length while the decrement is pending).

A bucket is empty if  bucket size <= Transaction Length, or <= maxPduSize + Transaction Length  (if a decrement pending)

When the Primary Bucket for a queue is considered empty, the secondary bucket will be used which will send traffic at a demoted priority. Once this secondary bucket is empty the queue cannot send any more traffic.

Credit Manager Parameters

 

The credit manager in the CTL initiator checks and manages the credits for controlling the request injection rate for the input queues. The types of credits supported are –
  1. Dedicated credits -  These credits are pre-allocated at initialization time (or can be re-initialized with credit re-sync message), and never need to be requested. The dedicated credits are per queue and are configured through parameters..
  2. Shared credits – These credits need to be requested when needed (R1 - not supported)
  3. No credits – There are no credit counters or checks

Credit units –
  1. Request Credits – A credit per read or write request
  2. Data credit – Credit for write data, the units are fixed (parametrized, crdDataUnit).

For a request to be eligible for arbitration, the request credit count has to be non-zero and additionally for write requests – data credits have to be above Delta, the normalized zero data credit.
  • Credit counters are per Target (parametrized) – mapped directly to one queue.
  • Credit counters are separate for data (write data) and requests (read or write)
  • Data credits can go below delta/normalized zero (delta is based on maxPduSz, below that is considered negative). This prevents holding off the write requests due to write credits fragmentation.
  • Credit counters can be for – dedicated credits only or shared credits only or both 
  • There can be QoS to credit threshold mapping

wCrdMngrReqCnt[int, int int, ...]An array of integers indicating the request dedicated counter width. Each entry in the array is assigned a queue. (0 is disabled)
wCrdMngrDataCnt[int, int int, ...]An array of integers indicating the data dedicated counter width. Each entry in the array is assigned a queue. (0 is disabled)

Timeout Description and Parameters

A timeout check for a request can result in an errored response to be generated. Timeout logic will set a bit for each request on each iteration of the timeout clock (each iteration is determined either determined through registers or timeoutErrCount). If on next iteration, the bit is still set, the request is marked to be timed-out. A response with error is generated (apart from interrupt set) and the request context is cleared out for the timed-out request. This is enabled by parameter (timeoutErrChk). 

  1. If the request was a write there will be a write command response.
  2. If the request was a read both a command response and read data will be sent back.

Parameter NamePossible ValuesDescription
timeoutErrChk<true or false>Enable timeout check.
timeoutErrCount<int>Timeout count in clock cycles.
timeoutUseExternalValue<true or false>Indicates whether count comes from registers.
