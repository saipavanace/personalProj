How are exclusive monitors implemented in DCE?DCE, AIU  

Current architecture does not provide much details on how exclusive monitors are supposed to work other than description of bits carried around. Arch document seems to indicate only coherent exclusives are supported, need clarification if non coherent exclusive should also be supported.

 

Following is the current implementation for 3.0

 

CHI specifies Ex Load as ReadClean, ReadNotSharedDrity & ReadShared. These translate to CmdRdCln and CmdRdVld in Concerto. Ex Store is specified as CelanUnique which translates to CmdClnUnq in concerto.

On receiving Ex Load CHI-AIU always responds back with pass when data is transferred.

DCE implements a configurable number of (minimum one) tagged monitors. The tag monitor consists of a valid bit for each Aiu Funit ID and LpID pair, i.e. one bit for each processor in the system, and an associated cacheline address field. The valid bit for each monitor indicates whether a given processor is in the midst of performing an exclusive access sequence, while the cacheline address field captures, or tags, the cacheline address for a given sequence. A tagged monitor is in-use if one or more processors are performing an exclusive access on the tagged address, i.e. any valid bit is set in the tagged monitor, and available if no processor is performing an exclusive access on the tagged address, i.e. no valid bits are set.

Execution:

When DCE receives an Ex Load an address cam is done on all active tagged monitors and either one of the following three scenarios will get executed

  • If there is an address match corresponding processors ID valid bit is set in that monitor.
  • If there is no address match and an empty tagged monitor is available then that monitor is loaded with the address and the corresponding processors ID valid bit is set.
  • If there is no address match and no empty tagged monitor are available then nothing is done.

In addition when an Ex Load is received, any old Exclusive loads from the same processor (where the address is different) that are being tracked by the DCE will be cleared. This means at any given time a processor can do only one store per DCE (in other words address region specified by DCE interleaving).

When DCE receives an Ex Store and there is a match in the tagged monitors then that transaction is marked as pass and all other processors that were being tracked by that monitor are cleared. If there is no match Ex Store fails.

 

Note that this implementation restricts one exclusive operation per processor per address region. here the address region is defined by the interleaving done across DCEs.

Address Map Implementation   

Address map is implemented as described in 014 version of Ncore3SysArch document
(Chapter 4 Storage Architecture) with following exceptions:

Only CSR configurable Address map option will be implemented.
DMI grouping interleaving functions will support only 2 options per grouping selectable via CSRs
2 options each for interleave 2, 3 & 4

 

The dmiIGSV is a vector (i.e. array) of Interleave Group Set (IGS).Each Interleave Group Set contains one or more Interleave Groups (IG).Each Interleave Group has the number of DMIs, and the vector of the DMIIDs.

u.paramDefault('dmiIGSV', 'array',   

[ 
{IGV : [{DMIIDV:[0]}, {DMIIDV:[1,2]}, {DMIIDV:[3,4,5]}, {DMIIDV:[6,7,8,9]}] }, 
{IGV : [{DMIIDV:[0,1,2]}, {DMIIDV:[3,4,5,6]}, {DMIIDV:[7]}, {DMIIDV:[8,9]}] }
]);
 

The dmi2WIFV is a vector (i.e. array) of 2-Way Interleave Functions.

Each Interleave Function comprises an array of integer for primary bits, and an array of string for secondary bits.

u.paramDefault('dmi2WIFV', 'array',
[ 
{PrimaryBits : [7], SecondaryBits: ["'h10000000"]},
{PrimaryBits : [9], SecondaryBits: ["'h40000000"]}
]);

 

The dmi3WIFV is a vector (i.e. array) of 3-Way Interleave Functions.

Each Interleave Function comprises an array of integer for primary bits, and an array of string for secondary bits.

u.paramDefault('dmi3WIFV', 'array',
[
{PrimaryBits : [7,  8], SecondaryBits: ["'h10000000", "'h20000000"]},
{PrimaryBits : [9, 10], SecondaryBits: ["'h40000000", "'h80000000"]}
]);

 

The dmi4WIFV is a vector (i.e. array) of 4-Way Interleave Functions.

Each Interleave Function comprises an array of integer for primary bits, and an array of string for secondary bits.

 

u.paramDefault('dmi4WIFV', 'array',
[ 
{PrimaryBits : [7,  8], SecondaryBits: ["'h10000000", "'h20000000"]},
{PrimaryBits : [9, 10], SecondaryBits: ["'h40000000", "'h80000000"]}
]);
