Replacement policies:
Number of ways is restricted when p-LRU is selected => odd number not supported & 2, 4, 8, 16… numbers only supported. This affects Maestro GUI and MAES-6368 has been created. 

2.1.1 CHI-E commands:
Some commands are not supported, error shall be reported in case these commands are received ("no data" error shall be raised). The Ncore 3.6 Error architecture specification is going to be updated accordingly
 

2.5 DVM support:
DVM 8.4 is the new version but all the previous version shall be still supported: DVM 8.1 shall be texted by DV.
Any agent connected to Ncore units will have a register with the supported DVM version.
Michael created a architecture JIRA ticket to create a register to define the highest DVM supported version.
Then Maestro has a parameter as well to set the supported DVM version to drive the RTL implementation.
DV to run regression based on the register or the parameter value and verify that both are consistent
Unused DVM ports shall be tight to zero at top level because, on the unit, the highest level is supported and all the corresponding ports are always implemented.
2nd review held July-17th:
DVM parameters and tables have been reviewed.
In case a version is not supported, Maestro shall not allow to select the version (waiting AEs feedback)
DVM version is a User's settable parameter at configuration time: it is a default value then read only register (in the JSON file)

JIRA ticket exists for Arch and Maestro support: CONC-12037/ MAES-6183

  • use Tenstorrent configuration to verify the bandwidth on 1) models and on 2) RTL with 4th network

Ncore connectivity architecture specification review:
Mainly a Maestro change, update of JSON parameters passed to the units
There is a parameter in Maestro that selects between 2 to 4 control networks: 1 network is not supported because of the risk of deadlock.
The tables define which messages are passed to which network.
When less than 4 networks are available, messages are combined as specified in Chapter 1.2.
With 3 networks, 2/3 of the theoretical bandwidth will be achieved while with the 4th network, the full theoretical bandwidth should be achieved.
There is no specific rules for Maestro in term of connectivity to the units. 

Supplemental Architecture specification review:
RB ID management chapter has been added that is more a micro-architecture implementation.
There is no software change associated with these new messages
HW design team will change logic in DMI and DCE blocks

  • Confirm the use case for for the secure/ non secure/ boot remapping Bernard Bonardi

  • Create a signal that controls the secure/ non secure mode of the registers + secure mode property added to the registers. This signal is controlled at the SoC level and it is up to the user to set the correct mode according to its CPU/ Application (note that with arm CPU, secure register is the default)

  • Boot is definitively non coherent

  • The remapping of the registers map in mission mode should only happen at the end of the boot sequence (+ remapping of the boot address that is done frequently). This is useful in case of several Ncore in the same SOC.

APB4:
Secure or non-secure boot supported: boot should be secured
User's manual shall report how to boot the Ncore: scope to be discussed and it could be a Maestro feature in the future. 
Boot is not coherent because all the registers are not set yet.
When APB4 reads or writes to a wrong address, the CHI and IO-AIU Units just check that the address range is valid, the DMI and DII units check that the CSR address is correct. System gets a response error otherwise.
In Ncore 3.6, the CSR base address can be re-programmed: refer to JIRA ticket CONC-7448
