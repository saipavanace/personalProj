Types of virtual network usage - decided up front, added to improve throughput, other?

Syed Shah (Deactivated)

Sanjay Deshpande (Deactivated)

Cases described:

  1. Hypervisor configures virtual networks at boot time
    We need to export a file to describe connectivity, virtual channel info & more - currently there is no standard format. Purpose: supply information for the user who setting up the virtual channel configuration for the hypervisor.
    This is for a multi-use configurable chip.
  2. Virtual networks chosen at pre-RTL NOC design time.
    This is for a chip for dedicated use with preconfigured virtual channels or no virtual channels. The no-virtual-channel case describes all of the current FlexNoC / Ncore chips.
  3. Virtual networks/channels for QoS - priority / Service Level Agreements (SLA)
  4. Virtual channels added during topology design to fix problems
    1. Add virtual channel at pre-RTL NOC design time to relieve throughput bottleneck.
    2. Virtual channels to relieve circular dependencies.
 

How does the user describe a virtual network?

  • does he need to specify #channels on a per-link basis?
alldesigner needs to specify a lot of info for each link
  • number and names/ids of virtual channels
  • QoS level for each virtual channel
  • depth of queue for each virtual channel
  • ... lots more, to be specified

Can we simplify his work, at least in case #2, above? If the user needs to specify and name each channel on each switch manually, that could be a lot of work across hundreds of switches.

Maybe have him specify a virtual network's connectivity and automatically create channels such that each virtual network will have a dedicated channel through each switch, and let the user then coalesce channels where he wants. Alternatively, just create a single virtual channel through all switches and let the user create extra channels as needed?

 How do we portray virtual networks to the user? we didn't have time to address this
