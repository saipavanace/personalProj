It is a generic hardware library component module that multiplexes Concerto message TX ports to assigned SMI TX ports, and demultiplexes from assigned SMI RX ports to Concerto message RX ports.

TACHL Parameters

wFPortId - width of FPortId (for Ncore3, Maestro always sets it to 1, and FPortId is always tied to 1'b0).

wFUnitId - width of FUnitId

smiTxPortInterfaces - an array of SMI TX Ports interfaces

smiRxPortInterfaces - an array of SMI RX Ports interfaces

concertoTxInterfaces - an array of Concerto Message TX interfaces

concertoRxInterfaces - an array of Concerto Message RX interfaces

smiHeaderSignalsToProtect - an array of SMI header signals to protect

concHeaderSignals - an array of Concerto Message header signals

concDataSignals - an array of Concerto Message data signals

Concerto to SMI Bit Field Mapping

Concerto Message Header bit fields (defined in ConcMsgHdr.packet.cpr as an array) are passed on to the corresponding SMI packet bit fields:

SMI ndp_targ_id = Concerto target_id

SMI ndp_src_id = Concerto initiator_id

SMI ndp_msg_id = Concerto message_id

SMI ndp_msg_type = Concerto cm_type

SMI ndp_msg_pri = Concerto priority


Concerto Message Body bit fields (defined in ConcMsgBody*.packet.cpr as an array) are packed into SMI ndp_ndp field, with
the first array element at the most significant bit position,
the last array element at the least significant bit position.

Question: FPortId and Steering?


From: John Coddington <John.Coddington@arteris.com> 
Sent: Friday, June 17, 2022 9:25 AM
To: Boon Chuan <Boon.Chuan@arteris.com>
Cc: Mohammed Khaleeluddin <Mohammed.Khaleeluddin@arteris.com>
Subject: Re: FPortId and Steering

Boon,

What I remember from discussions with MK and Sanjay, the portID was needed in the case where a block had multiple inputs or outputs into a single network. Using this you could always assign unique portIDs to all RX SMI ports (or TX SMI ports), and then the routing would always work, regardless of how you hooked them up to the networks. You can validate this with MK.

These two IDs get concatenated together and are then passed as a single ID on the SMI interface to be used by the packetizer to generate the route.

If steer exists in the SMI interface, it is concatenated with ID (which itself is a concatenation of FUnitId and FPortId) and used to generate the route. In discussions with Sanjay, he said it could be used in the case where you wanted different message types to follow a different paths through a single network between the same initiator and target. It has never been used, even though there have been times steer was in the SMI signal list. In those cases steer was always set to all 0s. Steer never made it into the packet (but there were discussions that if you had hierarchical routing it would need to be added if used.)

John

From: Eric Taylor <Eric.Taylor@arteris.com>

The user field contains a mini user field for every 64 bits.

So for example lets say we have a 128bit bus the dp_user filed will be packet like so.

dp_user = {
                          dw1.user,
                          dw0.user
                    }

Where dw*.user = {
                         dbad, (1 bit)
                         dw_id, (3 bits)
                         dprot, (8 bits if ECC, 1 if Parity)
                         be (8bits)
              }

So overall dp_user = {
                         dw1_dbad,  
                         dw1_dw_id, 
                         dw1_dprot,
                         dw1_be,
                         dw0_dbad,  
                         dw0_dw_id, 
                         dw0_dprot,
                         dw0_be 
                  }
