Symphony Instructions:

This current method of running zero-wire-load synthesis on a symphony based environment requires having to first build a symphony environment, then choose a configuration that is currently available with the appropriate scripts. 

Build symphony environment: 

Create a work area using the instructions in the link below to create an environment that will be used to run zero-wire-load synthesis. Follow instructions #1-4. 

One for All Simulation - Regression Tool - "runsim"

Now you can make any adjustments to the TACHL code in hw-sym/rtl/<block>/src/ directories to test any enhancements or updates. 

We currently have a generated 4x4 system with various configurations in each ATU to use to run synthesis on. (sym_synth_4x4_512e_256e). It has different size of context entries & features in each ATU. 


Build local maestro image to pick up your latest workspace TACHL updates:

(When any new edits to hw-sym/hw-lib are pulled or edited, you must re-run these two commands)

rm -rf $WORK_TOP/maestro_image; build_custom_product_image.js -hws $WORK_TOP/../../hw-sym -hwl $WORK_TOP/../../hw-lib -bd $WORK_TOP/maestro_image

setup_maestro --build_dir $WORK_TOP/maestro_image

Run the following commands to setup the environment to get ready to run zero-wire-load synthesis: 

cd $WORK_TOP
mkdir synthesis_sym_synth_4x4_512e_256e
cd synthesis_sym_synth_4x4_512e_256e
cp $WORK_TOP/../rtl/lib/bringup/sym_synth_4x4_512e_256e/*.tcl .

Run maestro to generate RTL (use stable version # indicated on -mbn option): 

run_maestro -t run.tcl -mbn $MAES_BUILD -sdj -utd -hws $WORK_TOP/../ -g


Run aria to generate synthesis directories and scripts:

node /home/boon/run_zero_wireload.js

This is the contents of the Aria script run_zero_wireload.js :

var {genDcSynScript} = require('/engr/dev/releases/utils/aria/latest/aria-linux-x64/resources/app/index');
genDcSynScript({
topDesign:'gen_wrapper',
cprInterfaceDirs: `${process.env.REPO_PATH}/assets/interface`,
techNode: 'tsmc7',
topoMode: false,
incrementalOpt: true,
clockUncertainty: 0.20,
compileCommand: 'compile_ultra -no_boundary_optimization -gated_clock',
bottomUpSynthesis: true
});


Now you have the ability to run synthesis on a specific block within this configuration. 

Either atui, atut, request or response network, for example. Look at the available blocks 'dc_scripts_*'. Ensure the logic/feature you want is included in the block you choose. I choose atut_axi_d here: 


cd output/synthesis/synopsys/scripts/atut_axi_d



Run the following command to run synthesis on the grid:

qsub -V -cwd -b y /engr/dev/tools/synopsys/syn_vO-2018.06-SP3/syn/O-2018.06-SP3/bin/dc_shell-t -64bit -f dc.tcl

Examine timing reports/results within the 'reports' directory and the log/errors will be placed in : dc_shell-t.o<ID_num>/ dc_shell-t.o<ID_num>


**NOTE**: If you make updates to TACHL or json and need to re-run synthesis, make sure to delete the following in the current directory you ran synthesis to help reduce corrupted databases.

rm -rf work/lib
rm -rf outputs/*

**NOTE**: Maestro/Aria synthesis flow assumes:
output/synthesis/synopsys/scripts are the synthesis scripts directory cd 
output/synthesis/synopsys/top are the synthesis run directory

According to the Maestro/Aria flow, user is supposed to
