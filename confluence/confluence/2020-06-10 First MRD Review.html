NCR3-MRD-3

Performance ModelAnkit Shah

Product shall create loosely timed, approximately timed and cycle accurate performance models. 

  1. run 10k-20k cycles per second
  2. Integrates with Synopsys PA
  3. Support instrumentation

Model team was told to focus on approximately timed. Cycle accurate was responsibilty of RTL team.

  1. hw_cfg_7 can run 2K cycles/sec
  2. using the standard TLM definitions, customer can theoretically build with PA or virtual platform of choice, but model team was told to avoid PA as a requirement
  3.  Plan for all instrumentation with previous customer input but still in development.

For approximate time model, product shall need bandwidth and latency tolerance to be less than 5%.  

Suggest aiming for 90% accuracy with well defined benchmarks or traffic sequences. Trying to reach an accuracy with any random sequence would be difficult. When I asked JP how accurate did model needed to be, he said the accuracy did not matter as much as the relative movement: that is, it didn't matter if the model gave the exact same result as RTL as long as a 10% change in RTL performance was reflected as a 10% change in model performance.

NCR3-MRD-113FastModel Shall support DataChirag GandhiThe FastModel shall support data transfer for a customer to run a software application

Status: Deferred

A functional model to develop software and a performance model for architectural analysis have competing requirements.

Model has built in capability to do limited data, using the architectural model in a software view. "Limited" means the data is passed through the network with the payload, but in this mode

  • there is no attempt to correlate timing
  • there is no modelling of the intermediate concerto messages
  • there is no modelling of CSRs in software mode or architectural mode
NCR3-MRD-195SystemC model and RTL correlation Kurt Shuler

Outputted SystemC models and RTL models shall have similar functional characteristics, configuration interfaces/methods, and outputted metrics (bandwidth, latency, etc.).


We want to make sure the SystemC models and RTL simulation have similar results and behavior given the same or similar stimuli. We also want to ensure that there is some similarity in data output formats and model controllability to allow a better user experience.

For inputs:

Models have a defined format for exact specification of input stimulus in a trace format or traffic profile format. Trace format is defined according to Synopsys STL, which customers are already using. Profiles can also be randomized. Models also have a method of specifying exact address config to mimick behavior of CSRs that control address space.

RTL has designed their own separate input format. The controls for defining address space CSRs in DV testbench is "all randomized in system verilog."

Not sure how to run correlation without the ability to run the exact same scenarios in RTL and DV testbenches, whether through trace format or traffic profile.


For outputs:

Models can try to match bandwidth/latency formats of RTL. A lot of other stats in ncore3 are defined to match ncore2 model, which was already designed to meet customer requirements.
