Constraint Handlers / Solvers and Derivation engines

  • Solves yes/no questions
  • Randomizer depends on this.
  • Multiple levels of these yes/no checks.
    • At the single parameter level
    • At the unit level
    • At the system level
  • High-Level Module Overview
    • Constraint/derivation engine
    • Dependency Manager
    • Views
    • Ues IDs, Query and Traversal, Set, Notify
    • Constraint failures send to Application
    • Application sends Eval / Check cmds
  • Agreed this will be used for all libDM data consistency.   
    • If we don't imagine constraints changing from release to release we should put them in a constraint solver.  If the constraints are stable from release to release then can be candidate to encapsulate in the data model.
  • Plan on coalescing parameter changes to reduce checking.
  • Architectural strong guideline
    • Principle is to want the data model to be more stable than the rest of the application
    • Data model defines the data and the functions that can operate on it
    • The actions are initiated by an application.
    • Since an application already needs to initiate the constraint checking, strong bias to use the external constraint solver integrated with the application, messagin and error system vs adding similar constraints in the DM classes themselves.
  • ID Aware Traversal functions will be provided by the DM.
  • One data model with multiple views.  Better to write your constraints in the view you're looking at it.
  • One constraint solver per view.
  • Decision - continues as the architectural framework for all constraint solvers.
  • Would like to see more use cases for other views (e.g. connectivity or topology)
  • Boost::Signal2 is thread-safe for implementing the notify.  Publish / Subscribe mechanism.
