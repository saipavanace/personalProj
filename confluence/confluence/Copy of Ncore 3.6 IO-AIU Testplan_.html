TB Components/Objects

IOAIU TB_Top

axi_if virtual interface is defined as a array of size N (N=nNativeInterfacePorts), each virtual interface handle points to the actual axi interface connected to each ioaiu_core. 

AXI4 Monitor

This monitor looks at the following interfaces between the AXI4 BFM (or one of the AXI4 VIPs) and the IOAIU DUT: -       AR -       R -       AW -       W -       B -       AC -       CR -       CD 

On the data channels, the monitor will collect all beats of data before sending the complete packet with all the data. There are also beat by beat checks but all data comparison is done on the last beat. 

SMI Monitor

The SMI monitor looks at the SMI master and slave request and response interfaces between the IOAIU and the system BFM.

CCP Monitor

CCP monitor looks at the interface between external logic and CCP.

IOAIU Scoreboard 

The goal is to change the IOAIU scoreboard as little as possible from 3.2 to 3.4 to accommodate multi-port. Instead of changing the scoreboard, we will change what it is connected to, for example, the smi connection.

For 3.4, each IOAIU scoreboard will receive SMI packets from a SMI demux output specifically for that core, not the SMI monitor itself. The IOAIU scoreboard and SMI monitor do not need to be multi-port aware, just the SMI demux.

The function write_ioaiu_smi_port() does not need to change for multiport env. The scoreboard will be getting SMI packets from the smi_demux component instead of the smi_monitor directly, but the scoreboard should not care about the UVM source of the packet.

We need to add a check to process_snp_req, called by write_ioaiu_smi_port(), to make sure that the snoop address of the SMI packet belongs to that core. This should report an error if the smi_demux sends a snoop packet to the wrong core.

IOAIU Scb Txn

ioaiu scb txn is an transient uvm_object, which tracks a complete transaction, during its life cycle in IOAIU block. A new ioaiu scb_txn object is created for a native write addr pkt, native read addr pkt, ccp evict pkt, and a SMI snoop pkt, since they mark the starting point of a transaction in IOAIU. This ioaiu_scb_txn object is then pushed into an outstanding ott_q in ioiau_scb. predictions for expected pkts are made in this class, based on the information from the initial packet, or subsequent packets. When expected packets arrive, they are compared and checked against the predicted pkts. When all the expected pkts arrive and checked, the ioaiu_scb_txn object is deleted from outstanding ott_q.

IOAIU Env

AXI Agent :

  • This component in the IOAIU is responsible to drive the AXI/ACE/ACE-Lite/ACE-Lite-E packets on the Native interface of the DUT
  • This takes in the AXI packet (axi_seq_item) driven from an axi_sequence and is converted to the respective pin toggles on the pins of the DUT

CCP Agent:

  • This is a passive agent (only monitor)
  • This components probes into the design to look at the CCP interface and construct packets going in and out of various CCP interfaces
  • The packets are then routed to the scoreboard

Scoreboard:

  • This component is responsible to determine the validity of the design by evaluating the packets flowing in and out of the DUT
  • Information is routed to the scoreboard from various analysis ports 
  • Native interface (AXI) packets built by the AXI agent, CCP related packets build by CCP agents and SMI channel packets SMI agent (living outside the IOAIU env) are routed into the scoreboard through various analysis ports.
  • All transactions are pushed in the outstanding ott_q and removed from ott_q, when all the expected packets arrive and contents verified. At EOT, it is expected that ott_q is empty.

Changes to ioaiu Env:

To accommodate multiport IOAIU verification, the following changes are adopted to the existing IOAIU Env.

  • The SMI agent is pulled out of the environment and now resides in a top wrapper env that is described in next section.
  • The APB agent is pulled out of the individual env to the wrapper env since the CSR address region is shared across multiple cores (TODO : Need to confirm this)
  • The q_channel agent is also pulled out of the individual env along with the probe_agent into the multiport env

IOAIU Multi-port Env

Ncore 3.4 adds support for multi-ported IOAIU with a ioaiu_core_wrapper that can instantiate multiple IOAIU cores. Single-ported IOAIU configs, such as those used in Ncore 3.2 will be structured in the RTL like a multi-ported IOAIU with 1 core.

There are significant IOAIU design changes related to multiport.

  1. Creation of new ioaiu_core_wrapper that instantiates N x ioaiu_core modules (1 per core).
  2. SMI is shared across the multiport ioaiu_core_wrapper instead of one per core. There is a CMUX module to handle this.
  3. Going into ioaiu_core_wrapper is a single shared APB interface. There is an APB_DMUX module that uses upper address bits provide a private APB interface for each core.
  4. Pma_slave is shared across multiport cores instead of one per core.
  5. Trace_capture logic has move out of ioaiu_core and into ioaiu_core_wrapper, shared across cores.

SMI demux

In single-ported 3.2, there was one set of SMI ports and the SMI monitor analysis port connected to the IOAIU scoreboard. In 3.4, with one set of SMI ports shared by multiple cores, we need something to route SMI packets from the SMI monitor to the IOAIU scoreboard associated with the correct core for that packet. The new SMI demux UVM component will take SMI packets seen by the SMI monitor and route them to one of several possible per-core analysis ports. Each IOAIU core scoreboard will connect to the appropriate per-core analysis port. The SMI demux component needs the intelligence to determine which IOAIU core's scoreboard to send each SMI packet to.

Note: RTL smi demux has logic to route the incoming SMI packet to IOAIU to the right core

DV smi demux has logic to route the incoming and outgoing SMI packet to the right core's IOAIU scoreboard for predicting/checking expected behavior.

APB demux

In single-ported 3.2, there was one APB interface per core. In 3.4, there is still one APB interface per core, but the one wider-address APB interface per multi-ported core_wrapper.

If there are eight cores, there will be a common APB interface going into the APB demux with 15 address bits. The APB demux will use the upper address bits to route the APB transaction to the appropriate core.

DV will have a similar situation. There will be a single APB agent with driver and monitor for the APB interface shared across the cores. If necessary, we will put extra APB monitors on the APB interface going into each core.

The APB demux cannot send the unmodifed full width APB address to each core, it must send a 12-bit address to the core.

Block Level Stimulus

AXI/ACE sequence

  • On each core's AXI4 interface, this sequence forks off random coherent and noncoherent read and write transactions. On ACE interface, this also includes Clean, Make and Evict transactions. 

  • By default, a random traffic sequence will be running, and transaction types, address repetition is determined by random knobs, that can be controlled at runtime.

  • For more directed testing, the default random traffic sequence can be overridden by another user-defined sequence where stimulus is very much controlled. Examples include- read_after_write_seq, eviction_seq, back2back_read_seq, snpdtr_seq etc.

system_bfm sequence

  • On SMI interface, this sequence forks off SMI traffic, with two roles -  as an initiator/master for SMI snoop traffic, and as a responder/slave for SMI response traffic(CMDrsp, STRreq, DTRreq etc).

reset sequence

  • This is a CSR sequence that is run to set some registers/CSRs when the system is out of reset. Most importantly this will include configuring the GPRAR registers.

attach sequence

  • This sequence sets some CSR registers to attach our IOAIU agent to the coherency domain. This would be needed for an IOAIU agent that gets snoop traffic

csr sequence

  • csr sequences are defined to run CSR traffic, based on different scenarios - like attach_detach_reattach, error registers setup and checks, no_addr_hit/multi_addr_hit, trans_actv_seq etc.

ioaiu virtual sequence

  • ioaiu_unit_test forks off the ioaiu_virtual_sequence. 

  • independent N axi_seqs are forked off, where N=nNativeInterfacePorts. Each axi_seq is run on axi_master_agent that is defined each ioaiu_env.
