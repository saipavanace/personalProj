We need to add a capability for customers to add their own names at the boundary of our design. To do this the following proposal to extend the present Interface definition in TACHL is proposed:

Original Structure:

{
  name: “a_”,
  params: {a bunch of parameters},
  interface: “a_string”
}

New Structure:

{
 name: “a_”,
 params: {a bunch of params},
 interface:  “a_string”
 synonymsOn: boolean,
 synonyms: {
    intSignalName0: [{name: extSigName0_0, width: 5},{name: estSigName0_1, width: 3}],
    intSignalName1:[{name: extSignName1_0: width: 1}]
 }
}

How TACHL responds when synonymsOn = true to an interface object passed to the nInterfaces function:

The thing to recognize is synonymsOn can only be true at the top instance of gen_wrapper.

When TACHL encounters an interface with synonymsOn = true, when it generates the ports for the module it will use the external names for the ports:
        input [4:0] extSigName0_0;
        input [2:0] extSigName0_1;
        input       extSignName1_0;
And it will automatic create the wire definitions and assigns to make it look internally like a normal interface:
     wire [7:0] a_intSigName0;
     wire       a_intSigName1;
 
    assign a_intSigName0 = {extSigName0_0,extSigName0_1};
    assign a_intSigName1 = extSigName1;
